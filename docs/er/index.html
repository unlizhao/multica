① 身份与工作区

用户、工作区、成员、邀请、令牌的关系。workspace_id 是 Multica 几乎所有业务表的"租户键"。

关键约束

① workspaces.slug 全局唯一,handler/reserved_slugs.json 黑名单防止与系统路由冲突; ② members 是用户和工作区的多对多桥,业务上唯一定义"谁在哪个工作区"; ③ verification_codes(图未画)是登录验证用的全局表,无 workspace_id; ④ 几乎所有其它表的访问控制起点是 middleware/workspace.go 解析 slug → 注入 workspace_id。

② Issue 域(协作核心)

issue 与其评论、标签、订阅、反应、附件、活动、依赖、项目的全部关系。多态字段(assignee_type / author_type / actor_type / uploader_type)统一用 {user, agent} 表达。

⚠ 设计注意

① issues.status 是固定枚举(backlog/triage/todo/in_progress/done/cancelled),无可配置流程引擎,这是与飞书 MeeGo 的核心差异点 —— 详见 迁移调研文档; ② comments.parent_id 自引用支持嵌套评论 + 解决线程; ③ attachments 同时挂在 issue / comment / chat_session / chat_message,通过非空字段判断归属; ④ activity_logs.details 是 JSONB,结构按 action 分支变化,用 format-activity 在前端格式化。

③ Agent / Task 域(Multica 的"心脏")

这是把 Multica 区别于普通项目管理软件的关键数据模型。agent_task_queue 承载 agent 任务全生命周期,task_messages 是 agent 工具调用流水,task_usage* 三张表组成 token 计量管道。

关键设计

① agent_task_queue 同时被 3 个来源触发:issue 指派(trigger_comment_id)、chat 对话(chat_session_id)、autopilot 自动开工(autopilot_run_id),三者互斥(应用层保证);
② parent_task_id + is_leader_task 支撑 squad 内"leader 任务路由到 member 任务"的二级结构;
③ Token 计量是三段管道:实时写 task_usage → dirty 表打标 → 定时任务把 dirty bucket 重算到 task_usage_hourly(以减小高频任务的写放大);
④ skills 与 agent_skill 多对多;skill_files 让一个 skill 可以承载多文件(如多步骤 prompt + 模板)。

④ Squad(Agent 团队)

把多个 agent 组成"团队",由 leader 负责把任务路由到合适的 member。squad_members.member_type 多态,目前只用 agent,但保留扩展空间。

外部表的 FK: leader_id → agents · creator_id/archived_by → users

⑤ Autopilot(规则触发自动开工)

四张表的"触发器 → 运行 → 投递"管道:autopilots(规则) · autopilot_triggers(订阅源) · autopilot_runs(每次触发) · webhook_deliveries(外部 webhook 落库)。

设计要点

autopilot_runs.task_id 接回 Agent / Task 域,把"规则触发"和"agent 干活"无缝衔接; webhook_deliveries.dedupe_key + replayed_from_delivery_id 支撑去重和重放; signing_secret 与 signature_status 用于校验 GitHub / Gitlab 等外部 webhook 真实性。

⑥ Chat(与 agent 的对话)

Chat 是除 issue 之外触发 agent 任务的第二条入口。一个 chat session 内,每条用户消息可派生一个 agent task,task 完成产出助手消息。

⑦ Inbox / Pin / Notification

用户旅程的"通知 + 收藏"三表组。inbox_items 是核心通知聚合;pinned_items 多态收藏 issue/view 等;notification_preferences 已在身份域出现过。

⑧ GitHub 集成

记录 GitHub App 安装、PR 元数据与 CI 状态,并通过 issue_pull_requests 把 PR 反向关联回 multica 的 issue。

⑩ 跨域全景:谁是中心节点

把 9 个域的"中心实体"拉到一张图,可以看到 workspaces 是租户中枢,issues 是协作中枢,agent_task_queue 是 agent 业务的中枢,三者合起来覆盖 80% 的查询路径。

⑪ 设计约定速记

约定含义示例
UUID 主键所有 PK 都是 pgtype.UUID,客户端可生成,便于离线;所有表的 id
workspace_id 租户键除身份/全局表外,所有表都带 workspace_id,middleware 强制注入issues, agents, autopilots, ...
多态字段对(*_type, *_id)Go 端用 {user, agent, ...} 字符串;无外键约束,由 service 层校验assignee_{type,id} · actor_{type,id} · author_{type,id}
时间字段统一 timestamptz;数据落库一律 UTC,渲染时按用户 users.timezone 转换created_at · updated_at · *_at
JSONB 弹性字段用于非结构化或快速演进的属性,新增字段不需要改 schemasettings · runtime_config · context · details · metadata
软删除 / 归档用 archived_at 时间戳代替删除,便于审计与恢复agents.archived_at · squads.archived_at
双向迁移每个迁移对应 NNN_xxx.up.sql + NNN_xxx.down.sql(80+ 个文件)server/migrations/
sqlc 生成访问层所有数据库访问经 db.Queries;手写 SQL 全部在 pkg/db/queries/*.sqlhandler/service 不允许直接写 SQL
