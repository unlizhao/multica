ApiClient · 前端唯一的 HTTP 入口

packages/core/api/client.ts

所有请求都从这里出去,所有响应都在 parseWithFallback 校验后才进入业务层。 这是 Web 与 Electron 的"出网网关",承担鉴权头、CSRF、请求 ID、错误结构化、401 处理等横切关注点。

核心签名

成员类型说明
baseUrl: string字段构造时传入,通常 NEXT_PUBLIC_API_URL 或 desktop 配置
token: string | null字段token 模式下保存 JWT;cookieAuth 模式下永远是 null

请求生命周期

关键设计点

1. cookieAuth vs token 双模式

同一个 ApiClient 同时服务 Web(cookieAuth=true,使用 HttpOnly cookie)与 Electron(cookieAuth=false,使用 Bearer token):

// Web 模式
new ApiClient({ baseUrl, cookieAuth: true });
// Desktop 模式
const c = new ApiClient({ baseUrl });
c.setToken(savedToken);

两种模式下 fetch 始终带 credentials: 'include',这样切换模式不需要修改 request 内部。

2. 结构化错误 ApiError

所有非 2xx 都被包成 ApiError:携带 status、code、message、requestId、原始 body。UI 层通过 err instanceof ApiError && err.status === 401 做精确分支,而不是字符串匹配。

3. 401 不直接登出

401 由 AuthStore 在 refreshMe/verifyCode 等关键路径上自行决策(避免短暂网络抖动把用户踢下线)。ApiClient 只如实抛出 ApiError。

4. 与 zod 解析的契约

每个领域方法都形如:

async listIssues(wsId, q) {
  const raw = await this.request("GET", `/api/workspaces/${wsId}/issues`, q);
  return parseWithFallback(IssuesResponseSchema, raw, []);
}

永远不返回 raw as T。这条规则在 02 · parseWithFallback 中详述。

调用关系

⚠ 容易踩的坑

不要在领域方法中"为了少写 schema"直接 return raw。 Multica 的 Electron 装机版会比 server 旧若干版本,字段漂移导致 #2143 / #2147 / #2192 三起白屏事故,所有都因为某处跳过了 parseWithFallback。
