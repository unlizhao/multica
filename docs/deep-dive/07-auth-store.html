AuthStore · 双模式认证的状态机

packages/core/auth/store.ts

Multica 不能"二选一"地支持 Web cookie 或 Desktop token —— 它必须同时跑通。 AuthStore 用一个开关 cookieAuth 让同一份逻辑覆盖两种模式,并在登出时严格清理工作区单例,避免上一账号的数据闪现给新账号。

状态结构

type AuthState = {
  user: User | null;
  isLoading: boolean;
  // actions
  bootstrap: () => Promise<void>;
  sendCode: (email: string) => Promise<void>;
  verifyCode: (email: string, code: string) => Promise<User>;
  loginWithGoogle: (code: string, redirectUri: string) => Promise<User>;
  loginWithToken: (token: string) => Promise<User>;  // 仅 desktop deeplink 用
  logout: () => void;
  setUser: (u: User) => void;
  refreshMe: () => Promise<void>;
}

双模式判定

logout 的清理顺序(必须严格按此顺序)

logout: () => {
  if (cookieAuth) {
    api.logout().catch(() => {});      // ① 让服务器清 HttpOnly cookie
  }
  storage.removeItem("multica_token");   // ② 清本地 token (token 模式)
  api.setToken(null);                  // ③ ApiClient 之后请求不带 Authorization
  setCurrentWorkspace(null, null);       // ④ 清空工作区单例 (见 05)
  resetAnalytics();                     // ⑤ 清埋点用户身份
  onLogout?.();                         // ⑥ app 层钩子 (跳路由 / 关 daemon token)
  set({ user: null });                  // ⑦ 最后翻 store,触发 React 卸载受保护页面
}

为什么是这个顺序

④ 必须在 ⑦ 之前 —— 如果先 set({ user: null }) 触发 React 卸载,而 setCurrentWorkspace 还没清,某些尚未卸载的组件会用旧 wsId 发出最后一次 query,响应的数据会被装到一个"未登录"的 cache key 里,下次登录如果不刷页就可能闪出陌生数据。

bootstrap:启动时如何"自我恢复"

关键设计:bootstrap 从不显示错误,失败就当作"未登录"。这样即使后端短暂 5xx,刷新两次后用户也能继续使用本地缓存(对 Electron 离线尤为重要)。

跨 store 的反向依赖

AuthStore 是为数不多直接 import setCurrentWorkspace 的 store。这种"高层依赖底层"的反向链,只允许出现在以下三处:

  • AuthStore.logout → 清工作区单例
  • workspace store.setCurrent → 调 setCurrentWorkspace
  • AuthBootstrap(CoreProvider 内的 effect) → 启动时联动两个 store

其它任何 store / hook 都通过 useSyncExternalStore(subscribeToCurrentWsId, getCurrentWsId) 只读访问。
