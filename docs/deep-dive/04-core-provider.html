CoreProvider · 应用启动装配的"主板"

packages/core/platform/core-provider.tsx

CoreProvider 是 Web、Desktop、(将来的)其他客户端共同的入口。它把 ApiClient、所有 Zustand stores、QueryClient、WSProvider、I18nProvider 一次性装配好, 并且用模块级单例保证只装配一次,即便组件树重渲染或 React Strict Mode 双调用。

Props 形状

type CoreProviderProps = {
  baseUrl: string;
  cookieAuth?: boolean;            // Web=true · Desktop=false
  storage: StorageAdapter;          // localStorage · electron-store · in-memory
  identity: { platform: "web" | "desktop" | "mobile"; version: string; os?: string };
  onLogin?: () => void;
  onLogout?: () => void;
  i18n?: I18nConfig;
  children: ReactNode;
}

注意:CoreProvider 不接受 ApiClient、stores、QueryClient 这些重对象 —— 它自己创建,以保证整个进程里这些对象唯一。

装配顺序

为什么是模块级单例,而不是 useMemo?

React 18+ 的 Strict Mode 会刻意双调用组件函数与 effect,以暴露副作用问题。如果 ApiClient / WS / QueryClient 用 useMemo 创建,有可能在某些重渲染场景被重新创建,导致:

  • WS 连接被重新建立,服务端短时看到两条连接
  • QueryClient 替换,所有 active query 失效
  • setToken 后短暂出现"半身份"状态

所以 CoreProvider 用一个简单的模块级 let core: CoreInstance | null = null,首次进入按 props 初始化、写入;后续直接复用。这天然契合 SPA 的"单例"假设。

SSR 注意

Next.js App Router 下,这个单例 只在客户端 起作用 —— RSC 与 server 渲染走不到 CoreProvider(它带 "use client")。所以不会污染请求间的状态。

与外部环境的契约

外部输入谁提供注意
baseUrlweb: NEXT_PUBLIC_API_URL · desktop: 用户在登录页输入切换需要重新初始化进程,运行时不允许变更
storageweb: BrowserStorage(localStorage) · desktop: ElectronStore · ssr: InMemoryStorage必须实现 get/set/remove/subscribe 接口
cookieAuthweb 必须 true,desktop 必须 false影响 AuthStore、ApiClient 与 WS 的握手方式
identityapp 启动时静态计算会被 ApiClient 加到 X-Client-* header,WS 也会带
onLogin / onLogoutapp 自定义web 用来跳转,desktop 用来同步 daemon token

消费者怎么用

// apps/web/components/web-providers.tsx
<CoreProvider
  baseUrl={baseUrl}
  cookieAuth
  storage={browserStorage}
  identity={{ platform: "web", version }}
  i18n={i18nConfig}
>
  {children}
</CoreProvider>

// apps/desktop/.../App.tsx
<CoreProvider
  baseUrl={cfg.serverUrl}
  storage={electronStorage}
  identity={{ platform: "desktop", version, os }}
  onLogin={() => daemonAPI.syncToken(token)}
  onLogout={() => daemonAPI.clearToken()}
>
  <RouterProvider router={router} />
</CoreProvider>

两端唯一的差异是 props,内部 Provider 树完全一致。这是为什么 packages/views 能在两边共享而无需任何条件分支。
