TaskService · Agent 任务的全生命周期

server/internal/service/task.go

TaskService 是 Multica 业务最重的一段代码:协调 issue、agent、runtime/daemon、消息流、token 用量、孤儿回收。 指派一个 agent 到一个 issue,从用户点击"分配"开始,到 agent 推完 push、改完 PR 状态、写完总结结束,每一步都经过这里。

从用户点击到任务结束的全链路

三个最值得记住的实现细节

1. ClaimTask 用 SKIP LOCKED 防"惊群"

SELECT * FROM tasks
WHERE status = 'queued' AND runtime_id = $1
ORDER BY priority DESC, created_at ASC
FOR UPDATE SKIP LOCKED
LIMIT 1;

多个 daemon 同时拉取同一队列时,Postgres 用行锁让每个 daemon 拿到不同的任务,无需在 Go 侧自己做分布式协调。

2. 一切 emit 都在事务提交后

所有 events.Bus.Emit 都包在 tx.AfterCommit(...) 里(辅助方法 postCommit)。如果事务回滚,WS 不会广播一个"虚假任务",避免客户端拉到一个不存在的 ID。

3. 孤儿回收基于"心跳 lease"

daemon 每 5 秒上报心跳,心跳里带 running tasks 列表。ReclaimOrphans cron 找:

  • 状态是 running 但 task.daemon_id 对应 daemon 的 last_seen > 60s
  • 状态是 claimed 但 last_seen > 30s

回收时记录 reason="daemon lost",并按 retry_strategy 决定 enqueue 重试或直接 fail。这条路径覆盖了"用户合上 Mac → 任务卡死"的场景。
