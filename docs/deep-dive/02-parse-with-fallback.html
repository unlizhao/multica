parseWithFallback · API 响应契约的最后防线

packages/core/api/schema.ts

装机版 Electron 客户端可能比服务器旧 6-12 个月。每个响应字段都是会漂移的合同。 parseWithFallback 把"信任服务器"变成"在客户端再校一遍",失败时降级,而不是把异常捅穿进 React 渲染。

核心签名

function parseWithFallback<T>(
  schema: ZodSchema<T>,
  raw: unknown,
  fallback: T,
  context?: string,
): T
参数说明
schema由 zod 描述的期望结构(api/schemas.ts 中按领域定义)
rawfetch 解析出来的原始 JSON,类型为 unknown
fallback校验失败时返回的安全值(空数组、空对象、合理默认)
context用于错误上报的端点标签,例如 "listIssues"

决策路径

为什么不直接 schema.parse(raw)?

zod 的 parse 在失败时 throw。在 React 19 的并发渲染下,如果它出现在 query function 里,React Query 会把这个错误传到 useQuery 的 error; 用户看到的就是带 ErrorBoundary 兜底的"出错了"页面。但事实上后端只是漏了一个新字段而已,UI 应该照常渲染。

parseWithFallback 用 safeParse 把这条路径改成:

  • schema 与 raw 完全匹配 → 返回类型化的 T
  • 不匹配 → console.warn + analytics 上报 + 返回 fallback,UI 继续渲染

典型用法

// schemas.ts
export const IssueSchema = z.object({
  id: z.string(),
  title: z.string(),
  status: z.enum(["backlog", "in_progress", "done"])
    .or(z.string()),  // 允许新枚举值降级为 string
  ...
});

// client.ts
async listIssues(wsId: string) {
  const raw = await this.request("GET", `/api/workspaces/${wsId}/issues`);
  return parseWithFallback(IssuesArraySchema, raw, [], "listIssues");
}

注意 enum 用 .or(z.string()):服务器后续新增了 "cancelled" 状态,旧客户端会降级为 string,switch 走 default 分支(灰色徽章),而不是崩溃。

三条金科玉律(写代码时记住)

  1. parse, don't cast:任何 as Issue / as IssueResponse 都是潜在 bug。
  2. fallback 必须能渲染:列表用 [],对象用合理默认,布尔用 false。空字符串 "" 比 undefined 更安全。
  3. switch 必须 default:任何 server-driven 的判断要假设会出现未知 case,UI 应降级为通用样式而不是空白。

真实事故复盘

#2143 — agent.kind 增加了 "copilot_cli",旧 desktop 客户端不在 enum 中,渲染崩。
#2147 — issue.status 从字符串改为对象 { key, label },旧客户端 status.toUpperCase() 抛 TypeError。
#2192 — autopilot.deliveries 字段在某条件下返回 null 而非 [],list.map 直接 throw。
三起事故的共同点:都跳过了 parseWithFallback。
