setCurrentWorkspace · 工作区上下文的全局单例

packages/core/platform/workspace-storage.ts

看似一个"setter",却是 Multica 避免跨工作区数据泄漏的核心机制。 Web 路由切换、Desktop tab 切换、登录登出,都通过它单点更新;所有 React Query key、persist store 命名空间、WS 订阅都基于它派生。

API

let _currentWsId: string | null = null;
const slugListeners = new Set<(slug: string | null) => void>();

React 接入点是 useSyncExternalStore(subscribeToCurrentWsId, getCurrentWsId),在任何组件树中"零配置"读到当前工作区。

数据流

为什么用 queueMicrotask?

React 19 严禁在渲染阶段调用其他组件的 setState。如果 setCurrentWorkspace 在 WorkspaceRouteLayout 的 useEffect 里同步触发监听者,而监听者是另一个组件的 useState setter,React 会抛 "Cannot update a component while rendering a different component"。

用 queueMicrotask 把派发延迟到当前调用栈结束后:

function setCurrentWorkspace(slug, wsId) {
  if (slug === _currentSlug && wsId === _currentWsId) return;  // no-op
  _currentSlug = slug;
  _currentWsId = wsId;
  queueMicrotask(() => {
    slugListeners.forEach(cb => cb(slug));
    wsIdListeners.forEach(cb => cb(wsId));
  });
}

这同时保证了同一个 tick 内多次调用只产生一次"最终值"派发(若实现了去抖,但目前是直接覆盖)。幂等检测(slug/wsId 没变就直接返回)是另一道保险,避免无意义的 invalidate 风暴。

命名空间存储

Zustand persist 的 key 不是固定字符串,而是 `ws:${wsId}:${name}`。这意味着:

  • 从 workspace A 切到 B 时,A 的 issue 过滤、视图模式等不会"染色" B。
  • 退出登录后再登录另一个账号,旧数据保留在原命名空间(下次回到同 wsId 自动恢复),不会污染新账号的数据。
  • 清理工作区时调一个 clearWorkspaceStorage(wsId) 就能精确清除。
