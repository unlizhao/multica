WSClient × realtime.Hub · 实时事件双方

packages/core/api/ws-client.ts · server/internal/realtime/hub.go

Multica 把"看板的实时性"做成原生体验:任何人在任何客户端的修改,毫秒级广播给所有订阅者。 前端 WSClient 负责 连接 / 重连 / 订阅 / 派发;后端 realtime.Hub 负责 授权 / room / 多节点中继。

整体拓扑

前端:WSClient

核心签名

成员说明
connect()建立 WebSocket。鉴权用 cookie 或 query token,带上 client identity(平台/版本/OS)
disconnect()主动关闭,清空重连定时器与 handler 集合
on(event, handler) → unsubscribe注册事件处理器,返回反向函数
onReconnect(cb) → unsubscribe重连成功后触发(用来触发 React Query refetch)
send(msg)发送主动订阅命令(subscribe_scope / unsubscribe_scope)

重连状态机

关键:onReconnect 不携带数据。它只是一个信号,React Query 收到后调用 queryClient.invalidateQueries() 重新拉取关心的数据,确保"断线期间错过的事件"被一次性补齐。

后端:realtime.Hub

关键结构

type Hub struct {
  rooms      map[uuid.UUID]map[*Client]struct{}  // workspace → clients
  scopeSubs  map[ScopeKey]map[*Client]struct{}  // (kind, id) 细粒度订阅
  register   chan *Client
  unregister chan *Client
  broadcast  chan Envelope                       // 待广播事件
  relay      RelayBackend                          // nil / RedisShardedRelay
  metrics    *Metrics
}

事件流

  1. HTTP 升级到 WS,认证后构造 *Client(含 userID、wsScope、send chan)。
  2. Client 写入 register,Hub 把它加入对应 room。
  3. 客户端发送 subscribe_scope {kind: "issue", id: "..."},Hub 在 scopeSubs 注册。
  4. handler 写库后调 events.Bus.Emit(ev) → Hub 把 ev 推到 broadcast 通道。
  5. Hub 的中央 goroutine 决定接收者:同 ws 的 room + scope 订阅者并集去重。
  6. 多节点部署下,relay 同时把 ev 写到 Redis Streams 的 shard,使其它节点 consume 后回放(每个节点对自己的 client 投递,Redis 起到节点间总线的作用)。

水平扩展:三种模式

REALTIME_RELAY_MODE行为用途
legacy不中继,本节点广播本节点 client单节点 / dev
sharded仅经 Redis Streams 中继。所有节点都从 stream 消费,自己节点产生的 ev 也通过 stream 派发生产多节点
dual本节点直接广播 + Redis 中继(双发,客户端去重)从 legacy 切到 sharded 的过渡期

为什么用 Streams 而不是 Pub/Sub

Redis Pub/Sub 没有消费组与持久化。Streams 用 consumer group 让每个节点独立消费、断线时不丢消息,代价是要做 ACK + trim;Multica 用按 wsId 哈希分片(shard count 可配),保证同工作区的事件顺序在单个 shard 内有序。

事件 → Query 失效的契约

这一段是整个前端架构的关键:WS 永不直接写 Zustand。流程是:

WS 事件
  → realtime/provider.tsx 的 useRealtimeSync
  → 每个领域的 ws-updaters.ts(issues/ws-updaters.ts ...)
  → queryClient.invalidateQueries(...)
  → React Query 自动 refetch
  → 组件重新渲染

这避免了"WS 给的字段集 ≠ HTTP 给的字段集"导致的视觉错位。所有数据真源永远来自 HTTP 拉取,WS 只是失效信号。
