目录结构与包用途

multica/ — pnpm workspaces + Turborepo + Go monorepo

Multica 是 pnpm workspaces + Turborepo 管理的 monorepo,加上一个独立的 Go 后端目录树。 本文给出仓库根的全部一级目录、Go 后端的所有子包、前端 packages 与 apps 的子目录,逐项注明用途、依赖方向、可见性边界。

① 仓库根目录

multica/ ├── apps/ # 五个可独立部署的应用 (web / desktop / mobile / docs) ├── packages/ # 跨应用共享的前端代码 (core / ui / views / tsconfig / eslint-config) ├── server/ # Go 单体后端 (HTTP + WS + CLI + daemon 全在这里) ├── docs/ # 项目级文档(html + md),你正在看的就是它的子目录 ├── e2e/ # Playwright 端到端测试,跨 web 与 server ├── scripts/ # 跨语言脚本(check.sh / dev.sh / install.{sh,ps1} 等) ├── docker/ # 容器入口脚本 ├── .github/ # CI / Issue 模板 / Release 工作流 ├── package.json # root + pnpm catalog,所有版本号集中托管 ├── pnpm-workspace.yaml # 声明 workspace 成员 (apps/* + packages/*) ├── turbo.json # Turborepo pipeline (build / typecheck / test / lint) ├── Makefile # 跨语言编排:make dev / make check / make test ├── docker-compose.yml # 本地一键起 PostgreSQL + Redis ├── .goreleaser.yml # Go CLI 发布配置 (multica CLI) └── ...
目录类型用途
apps/应用面向最终用户的可部署单元(各自有独立 package.json + 构建产物)。
packages/共享库跨 apps 复用的前端代码。不可独立部署,只通过 workspace 协议被 apps 引用。
server/服务Go monorepo 内"子 mod":单一 go.mod 同时提供 HTTP API、WebSocket、CLI、本地 daemon。
docs/文档项目级 HTML / Markdown 文档,与 apps/docs/(对外文档站)区分:这里是"项目内文档"。
e2e/测试Playwright 用例,需要同时启动 web + server,放在仓库根而非 apps/web 内。
scripts/工具shell + ps + mjs 混合脚本,提供启动 / 检查 / 安装能力,是 Makefile 的具体实现。
docker/部署Docker 镜像入口脚本(实际镜像构建在根目录的 Dockerfile{,.web})。
.github/CIGitHub Actions:ci.yml(typecheck/test) · desktop-smoke.yml · release.yml。

② apps/* — 五个应用

每个 app 都是独立的可构建单元。它们都依赖 @multica/core + @multica/ui + @multica/views(mobile 例外)。

目录技术栈用途与边界
apps/web Next.js 面向终端用户的主 Web 应用。Next 16 App Router。features/ 放 Web 独有逻辑(如 auth-cookie);platform/ 是唯一允许引用 next/* 的地方。所有页面壳子从 @multica/views 装载。
apps/desktop Electron 桌面端,electron-vite 构建。src/main/(主进程,自动更新等)src/preload/(隔离桥)src/renderer/(走 react-router-dom 而非 next)。共享 packages/* 以保持业务一致。
apps/mobile Expo / RN iOS 移动端。独立技术栈:Expo SDK 55 + RN 0.83 + NativeWind。只与后端共享 API,与 packages/* 仅共享类型(避免 RN 不支持的 DOM 依赖)。
apps/docs Fumadocs 对外产品文档站(MDX)。content/docs/ 双语 mdx;middleware.ts 处理 i18n。与项目内 docs/ 不同源。

③ packages/* — 共享库与边界铁律

四个共享包按"headless → 视觉 → 业务 → 配置"分层。包之间的依赖方向是单向的,违反即是 PR 拒绝项。

包类型用途与边界
packages/core headless 所有业务状态与服务端通信。Zustand stores · React Query queries/mutations · API client · WS client · 权限规则 · i18n。
禁止: react-dom · localStorage 直接调用 · process.env · next/*。所有平台依赖经 platform/ 注入。
packages/ui visual 原子视觉组件 (shadcn over Base UI) + Markdown 渲染 + tokens.css。
禁止:@multica/core(零业务,不知道 issue 长啥样)。
packages/views business 跨平台业务页面/区块组件:issues 看板、agent 列表、settings 面板等。
禁止:next/* · react-router-dom(路由通过 NavigationAdapter 注入)。
已国际化:locales/{en,zh-Hans}/*.json 25 个 namespace。
packages/tsconfig config 共享 tsconfig:base.json(strict 严格模式) + react-library.json(jsx + dom)。所有 packages 都 extends 它。
packages/eslint-config config 共享 ESLint flat config:base.js(通用) · next.js · react.js。

核心包 packages/core 的子目录(business 域 × 横切能力)

packages/core/ ├── api/ # HTTP + WS 客户端 + zod 防御 (ApiClient · WSClient · parseWithFallback) ├── auth/ # AuthStore + utils (cookie / token 双模式) ├── platform/ # 平台桥:storage / keyboard / core-provider (Web vs Desktop 注入点) ├── realtime/ # WS provider + use-realtime-sync (事件 → React Query invalidation) ├── i18n/ # i18next 装配 + locale picker ├── navigation/ # NavigationAdapter (抽象 next/router vs react-router) ├── permissions/ # 前端规则引擎 (rules.ts) ├── modals/ # 全局模态注册栈 ├── paths/ # URL 生成与 reserved slug ├── analytics/ # PostHog 埋点 ├── config/ · logger.ts # 全局配置与日志 ├── query-client.ts # TanStack Query 全局实例 │ │ # --- 业务域 (每域包含 queries/mutations/store/types) --- ├── issues/ agents/ autopilots/ chat/ squads/ ├── workspace/ projects/ runtimes/ skills/(在 runtimes 中) labels/ ├── inbox/ pins/ dashboard/ github/ onboarding/ ├── feedback/ └── types/ # 跨域共享类型 (api · events · activity · attachment ...)

视图包 packages/views 的子目录(每个对应一组页面)

packages/views/ ├── auth/ invitations/ invite/ onboarding/ layout/ # 装配 / 准入 ├── issues/ agents/ autopilots/ chat/ squads/ # 协作核心域 ├── runtimes/ skills/ projects/ labels/ members/ search/ # 协作辅助域 ├── inbox/ my-issues/ dashboard/ settings/ attachments/ # 用户旅程 ├── editor/ # Tiptap 富文本(mention/code/math/mermaid) ├── modals/ navigation/ platform/ i18n/ common/ # 横切 └── locales/ # 25 个双语 JSON + glossary.md

④ server/* — Go 后端

单一 Go module,但内部按"cmd/ 启动入口 / internal/ 私有实现 / pkg/ 可复用包 / migrations/ 数据库迁移 / db/queries/ sqlc 输入"五段式组织。

cmd/* — 三个二进制入口

目录产物用途
server/cmd/server服务主进程:HTTP + WebSocket。router.go 装配所有 handler;listeners.go 启动 Hub/Cron/Sweeper。
server/cmd/migrateCLI数据库迁移工具(up / down / version),嵌入 migrations/*.sql。
server/cmd/multicaCLI分发给最终用户的 multica 命令行:login / daemon / issue / agent / repo / runtime / setup / update。Cobra + GoReleaser。

internal/* — 不对外暴露的实现

包职责说明
internal/handlerHTTP 层所有 REST endpoint。一个文件 = 一个领域(issue.go / agent.go / autopilot.go ... 共 30+ 文件)。
internal/service业务层跨多 handler 的业务逻辑:task.go(任务编排) · autopilot.go · email.go · cron.go。
internal/realtimeWSWebSocket Hub + Redis 多节点 relay(详见 03-ws-client-and-hub)。
internal/daemonwsWS面向 multica CLI / 本地 daemon 的私有 WS 通道(任务下发 / 心跳)。
internal/daemon客户端本地 agent 执行器(被 CLI 调用):config / wakeup / health / gc / 自动更新 / execenv 沙箱。
internal/middleware中间件auth / workspace / ratelimit / csp / cloudfront / client / daemon_auth / request_logger。
internal/auth认证cookie · jwt · cloudfront 签名 · PAT cache · membership cache · daemon_token cache。
internal/events事件总线进程内 pub/sub:service publish,handler 与 hub subscribe。
internal/metrics监控Prometheus registry + db / http / daemonws / realtime 五类指标。
internal/storage存储对象存储抽象:local.go + s3.go。
internal/agenttmpl模板24 个内置 agent 模板 (bug-fixer / code-reviewer / prd-drafter / ...) JSON 加载。
internal/analytics埋点PostHog 客户端 + events 定义。
internal/cliCLI 助手multica CLI 内部:client / config / output / update。
internal/mention工具提及短码扩展 (@user · @agent)。
internal/util工具pgx 类型转换 · 文本工具 · 平台进程探测。
internal/logger日志统一 zap 包装。

pkg/* — 可复用包(对外可被其他二进制 import)

包职责说明
server/pkg/agentagent 适配每个 CLI agent 一个文件:claude · codex · copilot · cursor · gemini · hermes · kimi · kiro · openclaw · opencode · pi。统一接口 + 思考流解析 + stderr tail + 版本探测。
server/pkg/db数据访问queries/*.sql 是 sqlc 输入;generated/*.sql.go 是输出。所有数据库访问必须经此。
server/pkg/protocol协议WS 协议定义:events.go(领域事件)+ messages.go(传输信封)。前后端共享语义。
server/pkg/redact脱敏敏感字段(token / email / IP)统一脱敏。

migrations/ + sqlc.yaml

  • server/migrations/ — NNN_xxx.up.sql + NNN_xxx.down.sql 双向迁移,80+ 个文件,通过 cmd/migrate 与 internal/migrations(嵌入 fs)统一执行。
  • server/sqlc.yaml — 配置 sqlc:输入 pkg/db/queries/*.sql,输出 pkg/db/generated/。

目录用途
docs/项目内部文档:overview.html(总览) · deep-dive/(模块全景与深度解析) · research/(调研) · structure/(本页) · er/(数据模型 ER 图) · 多份 *.md(plan / rfc / outline)。

⑥ 包依赖速查(谁能 import 谁)

From → To允许?原因 / 例外
apps/web → packages/*✅Web 是装配端,可以引用所有 packages。
apps/desktop → packages/*✅同 web。注意 next/* API 不可用。
apps/mobile → packages/*⚠ 仅类型RN 缺 DOM 依赖,实际只 import @multica/core/types/*。
packages/views → packages/core / ui✅views 是业务装配层,需要业务态(core)+ 视觉(ui)。
packages/views → next/* · react-router-dom❌必须经 NavigationAdapter 注入。
packages/ui → packages/core❌ui 必须 headless,不知道业务。
packages/core → react-dom · localStorage · process.env · next/*❌core 是 headless,平台依赖只能从 platform/ 桥进来。
server/internal/* ↔ server/pkg/*✅ 单向internal 可以 import pkg;pkg 不能 import internal(否则被外部消费时会拉进私有代码)。
server/cmd/* → server/internal/* + server/pkg/*✅cmd 是装配层。
handler → service / db.Queries / hub✅handler 是 HTTP 适配,业务下沉到 service。
service / handler → db 直接 SQL❌必须经 sqlc 生成的 db.Queries。
