Agent 产物 (Artifact) 方案

docs/agent-artifacts/index.html

本文档拆解"agent 跑完一次到底产出了什么",分析现状痛点,并给出 agent_artifact 表 + daemon 提取器 + 评论回流 + prompt 注入 + UI 清单的完整 4 步方案。 每一步独立可上线、纯加法、零回归。

一次 agent 运行实际产出几类东西?

按产物地位从高到低排成 6 类:

#类别产物地位落库位置跨 agent 可复用例子
①代码 / 文件改动★★★★★ 终极产物git 仓库✅ 全员可见commit a1b2c3d
②评论 / 状态变更★★★★ 公共白板comments 表✅ 强制注入 prompt"已修复,原因 X"
③过程遥测 task_message★★ 实时观察task_message 表❌ 不读不复用tool_use(Edit, ...)
④Agent 私有内存★ 私有上下文jsonl (不在 Multica)❌ 仅同 agent 续聊~/.claude/.../*.jsonl
⑤用量代价— (代价不是产物)task_usage_*—cost=$0.18
⑥外部世界产物★★★★★ 用户视角的真产物第三方系统❌ 看不见,只能从 task_message 推断飞书 URL、生成的 PNG

关键洞察: ⑥ 在 MCP 时代是主流形态(写飞书/Notion/Slack/Jira/调内部 API), 但当前 Multica 没把它当一等公民建模 — 它只活在 task_message 的 tool/input/output 三个字段里,无索引、有 8KB 截断(daemon 端 L2890)、不跨 agent、 未来若引入归档策略则会失联。

飞书文档/生成图片/Slack 消息这些算产物吗?Multica 看得见吗?

算产物 — 是和 ① 代码 diff 同等地位的"用户视角真产物",但和 ① 有 4 个关键差别:

  1. 看不见产物本身:Multica 只看见"做这件事的过程" (tool_use.input + tool_result.output),真正的飞书文档在飞书服务器上,Multica 永远拉不到。
  2. 不可信:① git commit 不可篡改,⑥ 飞书 URL 只是 agent 报告它创建过 — 可能撒谎、可能 24h 后被删。
  3. 图片有特殊变种:
    • Write(file=docs/arch.png) 写到工作区 → 归 ① (进 git)
    • mcp__image_gen 调外部 API 生图 → 归 ⑥ (URL)
    • /tmp/foo.png 临时文件 → 既不是①也不是⑥ (过程性产物,daemon 退出即丢)
  4. 判断标准:这个东西明天还在不在?在哪里?谁能访问?

所以 ⑥ 想"持久 + 可追溯",必须借道 ② 评论 — 否则只剩 ③ 这条无索引通道。

注:task_message 是入库的(handler/daemon.go L1842 CreateTaskMessage), 关浏览器不会丢数据(刷新会补齐)。但它无索引、不参与续聊、 未来若引入归档策略则产物链接失联 — 这才是需要新建 artifact 表的真正原因。

档位 C — 可信引用(❌ 现在没有) — 需要新建 agent_artifact 表,见下方方案。

现状的 5 个具体痛点(P1-P5)
#痛点后果
P1没有索引 — 飞书 URL 散在 task_message.output text 里无法做"issue 维度的产物清单"
P28KB 截断(daemon 端 L2890,server 端无截断) — output = output[:8192]长 URL/长 JSON 末尾被切,偶发性丢链接
P3不入续聊上下文 — task_message 不参与 buildIssuePrompt换 agent 来 review 时看不到上轮飞书文档,跨 agent 协作断层
P4不入 ② 公共白板 — 产物只活在 task_message,未来引入归档策略则失联长期可追溯性 = 0
P5格式不统一 — 飞书是 JSON、DALL-E 是 URL、Bash 是文本没法写通用展示组件
解决方案总览(4 步,每步独立可上线) 
Step 1: 表结构 + 提取器(最小闭环)        <- MVP,纯加法,零回归
   |
   v
Step 2: 回流到 issue 评论(② 公共白板)    <- 让用户/agent 都能看到
   |
   v
Step 3: 注入 prompt(跨 agent 接力)       <- 让下一个 agent 自动读到上一轮产物
   |
   v
Step 4: 前端"产物清单"组件               <- issue 详情页可视化

设计原则(4 条铁律):

  1. 保持 task_message 不变 — 它是"过程录像",不要再叠功能。
  2. 新建 agent_artifact 表 — 让产物变成一等公民,和 task_message 解耦。
  3. 提取在 daemon 端做 — 避免 server 反向解析 task_message。
  4. 回流 ② 评论 + 注入 prompt — 跨 agent 可见、长期可追溯。在 FAQ: 换 agent 上下文 提案落地前,评论是唯一的跨 agent 通道;本方案落地后产物清单也成为通道之一,但仍借道评论走。
Step 1 — 表结构 agent_artifact

新建 migration(编号请核对最新,写本文档时为 107,落地前请重新核对避免撞车):

CREATE TABLE agent_artifact (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    workspace_id    UUID NOT NULL REFERENCES workspace(id) ON DELETE CASCADE,
    issue_id        UUID NOT NULL REFERENCES issue(id) ON DELETE CASCADE,
    task_id         UUID NOT NULL REFERENCES agent_task_queue(id) ON DELETE CASCADE,
    agent_id        UUID REFERENCES agent(id) ON DELETE SET NULL,

    kind            TEXT NOT NULL,
        -- 'file'    : 工作区内文件 (Write/Edit)
        -- 'url'     : 外部 URL (飞书/Notion/PR 链接)
        -- 'commit'  : git 提交 (hash)
        -- 'image'   : 图片 (URL 或 file,二级标记)
        -- 'message' : 外部消息 (Slack 消息 ID)
        -- 'unknown' : 兜底
    label           TEXT,                -- 人类可读名
    location        TEXT NOT NULL,       -- URL / file path / git hash
    mime_type       TEXT,
    metadata        JSONB,               -- 任意补充

    source_tool     TEXT NOT NULL,       -- 原始 tool name
    source_seq      INT,                 -- 对应 task_message.seq
    source_msg_id   UUID REFERENCES task_message(id) ON DELETE SET NULL,

    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),

    UNIQUE (task_id, source_tool, location)
);

CREATE INDEX idx_artifact_issue ON agent_artifact (issue_id, created_at DESC);
CREATE INDEX idx_artifact_task ON agent_artifact (task_id);
CREATE INDEX idx_artifact_kind ON agent_artifact (workspace_id, kind, created_at DESC);

设计要点:

  • 主键关联到 task_id + issue_id + workspace_id 三层,任意层级可聚合查询
  • metadata 用 jsonb,接新 MCP 不改 schema
  • UNIQUE (task_id, source_tool, location) 幂等:同 task 内不重复,跨 task 保留两条(两次独立产出)
  • 不挂在 task_message 上 — task_message 未来若归档不影响 artifact
  • kind 用 string 不用枚举 — 接新 MCP 时随时加新 kind,不改 schema

ORDER BY created_at DESC

-- name: ListArtifactsByTask :many

-- name: ListRecentArtifactsForPromptInjection :many

FROM agent_artifact
WHERE issue_id = $1
  AND kind <> 'file'
ORDER BY created_at DESC
LIMIT $2;
Step 1 — daemon 端提取器(artifact_extractor.go)

核心是中央分发器 + 按 tool name 路由:

type ArtifactCandidate struct {
    Kind       string         `json:"kind"`
    Label      string         `json:"label,omitempty"`
    Location   string         `json:"location"`
    MimeType   string         `json:"mime_type,omitempty"`
    Metadata   map[string]any `json:"metadata,omitempty"`
    SourceTool string         `json:"source_tool"`
    SourceSeq  int            `json:"source_seq"`
}

var urlRegex = regexp.MustCompile(`https?://[^\s"'<>)\]]+`)

func extractArtifacts(p toolPair) []ArtifactCandidate {
    if ext, ok := toolExtractors[p.Tool]; ok {
        return ext(p)
    }
    if strings.HasPrefix(p.Tool, "mcp__") {
        return extractFromMCP(p)
    }
    return nil
}

var toolExtractors = map[string]func(toolPair) []ArtifactCandidate{
    "Write":        extractFromFileWrite,
    "Edit":         extractFromFileEdit,
    "MultiEdit":    extractFromFileEdit,
    "NotebookEdit": extractFromFileEdit,
    "Bash":         extractFromBash,
    "WebFetch":     extractFromWebFetch,
}

4 个具体提取器:

  • extractFromFileWrite: 读 input.file_path, kind=file
  • extractFromBash: 白名单 git push/gh pr, regex 抠 commit hash / PR URL(避免噪音)
  • extractFromWebFetch: 读 input.url
  • extractFromMCP: 通配 mcp__ 前缀, 扫 output 里 URL, 图片后缀升级 image kind

关键点 1:提取器吃未截断的原始 msg.Output — 在 daemon.go L2890 截断之前取一份 rawOutput := msg.Output 给提取器用,再让原有 8KB 截断和 batch append 走原路径。 这样既不破坏 task_message 现有行为,又解决 P2。

关键点 2:mcp__ 双下划线前缀是 Claude Code 专有命名。 Codex / Cursor / Copilot 的 MCP tool 命名规范可能不同 — extractFromMCP 的判断条件需要按 backend 拆, 这属于 Step 1 落地时的"中风险"项。

Step 1 — 在 daemon.go 哪里插入提取调用

修改 daemon.go 的 case agent.MessageToolResult: 分支(L2873, truncation 在 L2890),伪代码:

case agent.MessageToolResult:
    // ... 维护 inFlightTools (现有逻辑) ...

    s := seq.Add(1)
    rawOutput := msg.Output  // ★ 截断前保留原文

    output := msg.Output
    if len(output) > 8192 {
        output = output[:8192]    // 给 task_message 用(不变)
    }

    // ... 现有逻辑:callIDToTool 查找、batch append TaskMessageData ...

    // 新增:提取产物(用 rawOutput)
    mu.Lock()
    inputForArtifact := callIDToInput[msg.CallID]  // ★ 需在 tool_use 分支记录
    mu.Unlock()

    if candidates := extractArtifacts(toolPair{
        Tool:   toolName,
        Input:  inputForArtifact,
        Output: rawOutput,     // 未截断
        Seq:    int(s),
    }); len(candidates) > 0 {
        mu.Lock()
        artifactBatch = append(artifactBatch, candidates...)
        mu.Unlock()
    }

额外改动:在 case agent.MessageToolUse: (L2855) 需同时记录 callIDToInput[msg.CallID] = msg.Input — 因为 tool_use 和 tool_result 分开两条消息, 提取时要把 input 和 output 配对。注意 pending text/thinking 的 flush 顺序不变, 保证 seq 先于 tool_result 入 batch。

Step 1 走完 → P1 (无索引) + P2 (截断) 解决。

Step 2 — 自动回流到 issue 评论(解决 P3 + P4)

任务完成时,server 端 CompleteTask 写评论前查一次 ListArtifactsByTask,把清单拼进评论 body:

**这次任务的产物:**
- 📄 [Q3 飞书报告](https://feishu.cn/docs/abc123) (飞书)
- 🖼️ [生成图](https://oai.../foo.png) (图片)
- 📄 `src/auth.go` (文件)
- 🔗 [PR #42](https://github.com/.../pull/42) (GitHub)

效果:产物从 ⑥ 升级到 ② (公共白板),未来归档 task_message 也不丢、跨 agent 必读 — 因为 buildIssuePrompt 本来就强制注入 comments。

幂等策略:仅在 server 端自动合成评论那条路径(agent 的 result.output 兜底)里追加产物清单; 如果 agent 已主动发了包含相关 URL 的评论,则不重复贴。 判断方法:查 comments 表最近一条是否已包含 artifact location,有则跳过。

Step 3 — 注入到下一个 agent 的 prompt

改造 prompt.go 的 buildIssuePrompt:

artifacts, _ := queries.ListRecentArtifactsForPromptInjection(ctx, issueID, 20)

if len(artifacts) > 0 {
    sb.WriteString("\n## Previous artifacts (from earlier agent runs)\n")
    sb.WriteString("Reference by URL/path; do not regenerate:\n\n")

几个反直觉的设计决策(值得 review)
  1. 为什么不在 server 端解析 task_message 来生成 artifact?
    跨节点、需要事务、性能差;daemon 端解析就近、便宜、不影响 server。
  2. 为什么 kind 用 string 而不是枚举?
    接新 MCP 时随时加新 kind,不改 schema。
  3. UNIQUE (task_id, source_tool, location) 会不会太严?
    不会。同 task 同 tool 写同一 URL 是"幂等同事件"; 跨 task 同 issue 写同一 URL 应该保留两条 — 那是两次独立产出。
  4. 要不要把 Bash 的 stdout 全部 URL 提取?
    不要。Bash 噪音大,必须靠 command 白名单 (git push / gh pr),否则污染产物表。
  5. 外部产物为什么必须借道评论(Step 2)?
    buildIssuePrompt 强制注入 comments — 这是跨 agent 唯一保底通道。 单纯写 artifact 表只解决"能查",没解决"下一个 agent 必读"。
一句话总结

现状:6 类产物中,⑥ (外部副作用) 只活在 task_message 的 tool/input/output 三个字段里 — 无索引、有 8KB 截断(daemon 端)、不跨 agent、未来若归档则失联。

方案:agent_artifact 表 + daemon 端 extractArtifacts 提取器 + 自动回流评论 + 注入下一轮 prompt + 前端清单组件 — 4 步独立可上线,每一步纯加法、零回归。

MVP (Step 1):1 个 migration + 1 个 sqlc query + 1 个 提取器 + 1 个 handler, 即可让"产物"成为系统一等公民。
