Session 管理 #

"session" 在 Multica 里承担三种完全不同的责任,刚好对应三种使用场景。 把它们混为一谈是最常见的认知陷阱。

轨道谁在用存在哪里形态
① 浏览器 session Web 前端 / Desktop (Electron) 浏览器 Cookie / Electron localStorage 短/中期 JWT
② CLI / daemon 登录会话 multica CLI、本地 daemon ~/.multica/config.json 长期 PAT (mul_) 或 daemon token (mdt_)
③ Agent 对话会话 (chat_session) 任何客户端 → 后端 → daemon → Claude Code Postgres chat_session 表 DB 行 + session_id 字符串作为 resume 指针

三轨之间的数据流

浏览器 ──登录──▶ Login handler ──发 JWT──▶ multica_auth Cookie ──▶ /api/* (Auth middleware)
                                                                       │
                                                                       ├─ 调 /api/tokens 建 PAT
                                                                       │           │
                                                                       │           ▼
                                                               ~/.multica/config.json
                                                                       │
                                 ┌─────────────────────────────────────┴───────────┐
                                 ▼                                                 ▼
                         multica CLI                                       multica daemon
                         (mul_xxx)                                         (mul_xxx 或 mdt_xxx)
                                 │                                                 │
                                 └─────────── /api/* (Auth) ──────────┐            │
                                                                      │            │
                                                                      ▼            ▼
                                                      /api/daemon/* (DaemonAuth + Redis cache)
                                                                      │
                                                                      ▼
                                                           chat_session.session_id
                                                           (Claude Code resume pointer)

常见问题

① 浏览器 session 是怎么管理的?Cookie 名、过期、续签都在哪?

一句话: 登录成功后,服务端用 auth.SetAuthCookies 一次写两枚 Cookie, HttpOnly 的 multica_auth 携带 JWT,可读的 multica_csrf 用作 CSRF 校验, 默认 30 天过期 (AUTH_TOKEN_TTL 可覆盖)。

关键代码:

  • 颁发: handler/auth.go 的 Login / Google Login → issueJWT(user)
  • 设置 cookie: auth/cookie.go SetAuthCookies
  • 注销: auth.ClearAuthCookies 把两枚 cookie 都设 MaxAge=-1
Cookie用途关键属性
multica_authHttpOnly,装 JWTHttpOnly + SameSite=Strict + Secure(仅 HTTPS)
multica_csrfJWT 派生的 HMAC非 HttpOnly,前端 JS 可读,需回传到 X-CSRF-Token
CloudFront-* 三件套CDN 私有资源访问由 cloudfront.go 同步签发

易错点

Secure 标志会跟随 FRONTEND_ORIGIN 的 scheme: HTTP 自托管场景 (LAN IP) 一旦把 Secure=true 强制设上,浏览器会静默丢弃 cookie,导致登录立刻失效。 cookie_test.go 里 TestSetAuthCookies_HTTPSelfHost 是为了固化这个行为。

① Web 和 Desktop 怎么区分 cookie 模式还是 token 模式?

一句话: 在前端 auth/store.ts 里通过 cookieAuth 选项区分,Web 走 cookie 模式 (浏览器自动带),Desktop / Electron 走 localStorage + Bearer 模式。

关键差异:

  • cookieAuth=true (Web): initialize() 直接调 api.getMe(), cookie 由浏览器自动附带;前端从来不接触 token 字符串。
  • cookieAuth=false (Desktop): 从 localStorage 读 multica_token, 手动塞 Authorization: Bearer ...;Desktop 没有共享 cookie 容器。

防御:为什么只在 401 时清 token

5xx / 网络断开 / abort 不清 token。这是被 #2143 / #2147 教训过的 —— 后端 rolling restart 时如果一抖就清,用户被无差别踢下线。 清 token 的责任由 ApiClient 的 onUnauthorized 回调专门承担。

② CLI 登录到底登的是什么?token 存哪儿?

一句话: CLI 登的是一个长期 PAT (mul_ 前缀),通过浏览器 OAuth-like 回调获取, 写到 ~/.multica/config.json;之后 CLI 和本地 daemon 共用这一份 token。

登录流程 (cmd_auth.go 的 runAuthLogin):

1. CLI 起一个本地 HTTP 监听 (loopback 或 LAN IP)
2. 浏览器打开 web app 的 cli_callback URL
3. 用户在浏览器里登录 (走 ① 浏览器 session)
4. 浏览器拿着 cookie 调 /api/tokens 创建一个 PAT (mul_xxx)
5. 浏览器把 PAT 推到 CLI 的本地回调
6. CLI 验证 GET /api/me 通了,写入 ~/.multica/config.json

配置文件 (cli/config.go):

// ~/.multica/config.json (默认 profile)
{
  "server_url":   "https://api.multica.ai",
  "app_url":      "https://multica.ai",
  "workspace_id": "uuid",       // 当前默认工作区
  "token":        "mul_xxxx"    // PAT
}

支持 profile 隔离:--profile staging → ~/.multica/profiles/staging/config.json。生产 / staging / 自托管可同时存在。

临时覆盖:CI 场景下用环境变量 MULTICA_TOKEN=mul_xxx,优先级高于配置文件。

② 三种 token (mul_ / mdt_ / JWT) 分别是什么?

一句话: 看前缀分流。auth/jwt.go 里 Generate*Token 各自负责生成。

前缀类型谁创建谁吊销用在哪
mul_ Personal Access Token (PAT) 用户:CLI multica login 自动建一条;或 web 设置页手建 用户撤销 / 过期 (DB 中心化吊销) CLI、daemon、第三方脚本
mdt_ Daemon Token 服务端在 daemon 注册时分配 daemon 注销 / 过期 (DB 中心化吊销) 仅 daemon ↔ 后端的高频心跳路径
(无前缀) JWT Login handler 颁发 仅靠过期,无中心化吊销 浏览器 cookie / Desktop bearer

PAT 和 daemon token 数据库里只存 SHA-256 hash (token_hash), 明文在签发时只露面一次;若用户丢失只能重新生成。

② 服务端怎么校验这三种 token?为什么有两套 middleware?

一句话: 普通 API 用 middleware/auth.go, daemon 专用通道 /api/daemon/* 用 daemon_auth.go。 两者都按 token 前缀分流,daemon 通道额外接 Redis 缓存来扛心跳压力。

Auth (普通 API)

优先级:

  1. Authorization: Bearer xxx (Desktop / 第三方调用)
  2. multica_auth HttpOnly cookie (浏览器)

走到 cookie 分支会额外验 CSRF (state-changing 方法必须带 X-CSRF-Token); 走 Bearer 不需要 CSRF,因为 PAT/JWT 不会被浏览器自动跨站发出。

DaemonAuth (/api/daemon/*)

按 token 前缀分流到三条路径:

  • mdt_ → 查 daemon_tokens 表 + Redis 缓存
  • mul_ → 查 personal_access_tokens 表 + Redis 缓存
  • 否则 → JWT 解析 (向后兼容旧 daemon)

为什么 daemon 通道单独有缓存

daemon 心跳约 15s 一次,缓存命中后 完全跳过 DB (包括 last_used_at 的 UPDATE 都跳了),把每请求一次 DB 折算成 每 10 分钟一次 (AuthCacheTTL)。这是性能要害; 副作用是 吊销延迟上限 = 缓存 TTL,最多 10 分钟。

② Daemon 怎么登录?为什么 multica daemon start 会报"not authenticated"?

一句话: Daemon 不会单独登录。它读取 CLI 写的同一份 ~/.multica/config.json,继承 PAT。所以必须先 multica login。

关键代码: daemon/daemon.go 的 resolveAuth() 读取 profile config,把 cfg.Token 注入到 daemon HTTP client。

报错原文:

not authenticated — run 'multica login' to authenticate, then restart the daemon

使用 profile 时:multica login --profile staging 之后再 multica daemon start --profile staging。

③ chat_session.id 和 chat_session.session_id 不是一回事吗?

一句话: 不是。id 是 Multica 自己分配的 UUID 主键 (Multica 内部对话 id); session_id 是 Claude Code 内部生成的 session id,用作 claude --resume <session_id> 的"续聊锚点"。

字段类型含义什么时候被填
chat_session.id UUID Multica 自己的对话主键 建会话时立即生成
chat_session.session_id TEXT (nullable) 外部 agent (Claude Code) 的 resume 句柄 第一次 agent 真正跑过之后才被回填

把 session_id IS NULL 当作"对话还没真正开过"的语义。

关键代码:

  • 033_chat.up.sql — 表结构
  • chat.sql 的 CreateChatSession / UpdateChatSessionSession / GetLastChatTaskSession
③ Chat 会话的"对话记忆"是怎么不丢的?

一句话: 三层防护:① UpdateChatSessionSession 用 COALESCE 写回, 空值不会覆盖已有指针;② 即使 chat_session.session_id 是 NULL,会走 GetLastChatTaskSession 从 agent_task_queue 捞最近一条带 session_id 的任务 (包括失败任务) 来 resume; ③ 删除前必须先取消 in-flight task,避免"答案没地方落"。

关键设计 (chat.sql 注释原文):

"even a failed task may have established a real agent session before failing, and we'd rather resume there than start over and lose conversation memory."

所以即使单次 agent 任务崩溃,只要它在崩溃前已经向后端报告过 session_id,下一条用户消息就能续上记忆。

删除流程(chat.sql DeleteChatSession 注释):

  1. LockChatSessionForDelete 锁住行
  2. CancelAgentTasksByChatSession 把 in-flight 任务取消
  3. DeleteChatSession 硬删,chat_message 走 FK CASCADE 一起删; agent_task_queue.chat_session_id 走 SET NULL 保留历史
跨议题:为什么 CSRF 只在 cookie 模式下生效,Bearer 不需要?

一句话: CSRF 攻击的前提是浏览器自动带 cookie 跨站发请求。Bearer token 必须由 JS 主动设置, 不会被跨站盗用,所以 PAT/Bearer 路径天然免疫。

这就是 middleware/auth.go 里 fromCookie && !auth.ValidateCSRF(r) 这条只在 cookie 分支启用的原因。 Desktop 走 Bearer,不需要前端额外维护 CSRF token。

跨议题:撤销一个 token 后多久真正失效?

取决于 token 类型和路径:

Token 类型吊销方式实际失效时延
JWT (浏览器 / Desktop)无中心化吊销= 剩余 TTL (默认 30 天)
PAT (mul_) — 普通 APIDB 标记 revoked立即 (Auth 中间件每次查 DB) 或 ≤ AuthCacheTTL (缓存命中)
PAT (mul_) — daemon 通道DB 标记 revoked≤ 10 分钟 (AuthCacheTTL)
Daemon token (mdt_)DB 删除≤ 10 分钟 (AuthCacheTTL)

所以 JWT 不能用作"安全敏感操作"的唯一凭据,例如改邮箱、改密码这类操作通常会再补一道密码 / 验证码确认。

把这些常被混淆的概念列一下
  1. "session" 三个含义:浏览器里的 session (① cookie 内 JWT)、 CLI 的 session (② PAT 写在 config.json)、chat_session (③ DB 行 + agent 内部 session_id) — 第三个跟登录无关。
  2. JWT vs PAT:看前缀。mul_ / mdt_ 走 token hash 查表; 其它走 JWT 解析。JWT 没有中心化吊销,PAT/Daemon token 可主动 revoke。
  3. CSRF 只对 cookie 路径生效:cookie 是浏览器自动带的,会被 CSRF 攻击; Bearer 必须 JS 主动设置,不会被跨站盗用。
  4. chat_session.session_id 和 chat_session.id 是两码事: 前者是 Claude Code 自己生成的 resume 句柄,后者是 Multica 自己分配的 UUID。 把 NULL session_id 当成"对话还没真正开过"的语义。
  5. Daemon 不会单独登录:它共用 CLI 写的那份 PAT;想让 daemon 跑必须先 multica login。

派单与上下文恢复 #

"我之前的 issue 能保证打到同一个 agent 上吗?如果离线了,会不会有别的 agent 接管?"

核心结论: 不是"看谁空就给谁",而是严格的赋值关系链: issue.assignee_id → agent.runtime_id → 派给特定 runtime。 enqueue 时 agent_task_queue.runtime_id 就被钉死,不是 claim 时再算。

三者的绑定关系

issue.assignee_id ──┐
                    ▼
                agent ──┐ agent.runtime_id (粘性绑定)
                        ▼
              ┌────────────────────────────────┐
 enqueue ───▶ │ agent_task_queue 行 (钉死)      │
              │  · agent_id   ← 永远是这个 agent │
              │  · runtime_id ← 永远是这台机器   │
              │  · session_id ← agent 自己写的   │
              │  · work_dir   ← 上次工作目录     │
              └────────────────────────────────┘

常见问题

能保证之前的 issue 打到同一个 agent 吗?

能,前提是 issue 的 assignee 没变。

证据来自 TaskService.enqueueIssueTask: enqueue 时按 issue.assignee_id → agent → agent.runtime_id 链路一次性把 runtime_id 写进 agent_task_queue,不会在 claim 时漂移。

额外的串行保证: ClaimAgentTask 语义上"同一 (issue, agent) 下,一次只允许一个 dispatched/running 任务", 但不同 agent 在同一 issue 上可以并行(为 squad / @mention 准备)。

场景行为
用户改 assignee之后的 task 走新 assignee,旧任务保留
Agent 被 archiveenqueue 时直接拒绝
用户点 "Rerun"仍是同一 agent,但带 force_fresh_session=true
评论里 @ 别的 agent走 mention path,新 agent 也会被拉进来跑
Squad 模式leader 派单后再派多个 worker 子任务,同 issue 多 agent
如果 runtime 离线,会不会有别的 agent 接管?

默认不会。任务和 runtime 是绑定的,离线即 fail,不会自动迁移。

检测路径(双保险):

  1. Heartbeat 主路:daemon ~15s 一次心跳更新 last_seen_at
  2. Sweeper 巡检路:周期调 MarkRuntimesOfflineByIDs, 把 last_seen_at < now() - stale_seconds 的行翻成 offline。 SQL 把 stale 谓词重复写进 UPDATE 的 WHERE 子句,避免 SELECT/UPDATE 之间被一次新心跳救活的 runtime 错杀

离线后: FailTasksForOfflineRuntimes 把 dispatched/running 的任务标 failed,failure_reason='runtime_offline'。 不会迁移到别的 runtime。

几个微妙的"看似接管"路径

(A) 同 runtime 重新上线 → RecoverOrphanedTasksForRuntime 把孤儿 fail,重新 claim queued。
(B) Stale dispatched 重新 claim → ReclaimStaleDispatchedTaskForRuntime 仅同一 runtime 才能重新拿(daemon_test.go 有专测)。
(C) Auto-retry → CreateRetryTask 新任务继承父 runtime_id,那台没回来就一直 queued。
(D) Agent 改绑 runtime → UpdateAgent 之后的新 task 落到新 runtime,但已 queued 的旧任务保留旧 runtime_id。
(E) Member 被踢 → CancelAgentTasksByRuntimeOrAgent 用 cancelled 而非 failed,让 daemon 优雅打断。

"上下文"在 Multica 里到底是什么?分了几层?
层级存在哪里用途
① Workspace 层workspace.context整个工作区的全局背景
② Agent 层agent.instructions / skills / MCP这个 agent 的人设、技能
③ 会话层agent_task_queue.session_id + work_dir续聊锚点 — 本议题核心

"上次 agent 跑过的对话记忆怎么找回来" 只关系到第 ③ 层。

session_id / work_dir 续上的规则是什么?

handler/daemon.go 里给 daemon 派任务的 response 返回 PriorSessionID / PriorWorkDir,具体策略:

情况session_idwork_dir
同 runtime + 非 comment 触发✅ 续✅ 续
不同 runtime(agent 改绑了机器)❌ 不续✅ 续(仍可保留 git checkout)
Comment 触发的后续任务❌ 不续(上下文常被 "Done." 类终结消息污染)✅ 续
用户点 "Rerun"(force_fresh_session=true)❌ 不续❌ 不续
哪些失败状态不能续聊?

GetLastTaskSession 的 WHERE 子句明确排除"中毒"类失败:

failure_reason NOT IN (
  'iteration_limit',           -- agent 自己跑到迭代上限
  'agent_fallback_message',    -- agent 输出了 fallback 终结句
  'api_invalid_request',       -- 400 invalid_request_error
  'codex_semantic_inactivity'  -- Codex 卡死状态
)
AND NOT (error LIKE '%400%' AND error LIKE '%invalid_request_error%')

这些 session 续上去会复现同样的卡死。失败任务也算 — 只要它在崩溃前已经把 session_id 报回过,就能续。

换 agent / @mention 后,新 agent 怎么获得上下文 #

一句话: 新 agent 不"知道"上下文,但它被 prompt 显式强制调 multica issue get 和 multica issue comment list 把整段历史读出来 —— Multica 故意设计的 "只信公开史料、不信私密 session" 模型。

     Issue 公共白板 (所有 agent 都能读)            
     ┌───────────────────────────────┐               
     │  · title / description         │              
     │  · comments[]                  │ ◀── multica issue comment list
     │  · status                      │ ◀── multica issue get        
     │  · linked PRs / attachments    │              
     │  · activity_logs               │              
     └──────────┬───────────┬────────┘               
                │           │                        
   ┌────────────▼─────┐   ┌─▼──────────────────┐    
   │ Agent Atlas      │   │ Agent Bob          │    
   │ Claude session A ✘ X ✘ Claude session B  │    
   │ (~/.claude/.../jsonl) │  (新建)            │    
   └──────────────────┘   └────────────────────┘    
        私密大脑(不互通)         私密大脑(不互通)
为什么不能继承上一个 agent 的 session?

GetLastTaskSession 的 key 是 (agent_id, issue_id),不是只看 issue_id。

把 assignee 从 Atlas 改成 Bob 后,Bob 第一次接这个 issue 时:

  • 查 (Bob.id, issue.id) → 一条都没有
  • prior.SessionID 为空 → daemon 不会透传 --resume
  • Bob 启动一个全新的 Claude Code session

这是有意为之:Atlas 的 JSONL 历史物理上保存在 Atlas 那台 runtime 的 ~/.claude/projects/...,Bob 可能在另一台机器、另一个 provider、另一个 work_dir。 硬续必失败。

那 Bob 怎么"知道"前面发生了什么?

Multica 在 prompt 层把"读公共历史"做成硬性指令。 prompt.go 与 runtime_config.go 注入到 CLAUDE.md / AGENTS.md 的关键句:

"Run multica issue comment list <id> --output json to read the full comment history — this is mandatory, not optional. Earlier comments often carry context the issue body lacks (... the prior agent's findings, the reason the issue was reassigned to you). Skipping this step is the most common cause of agents acting on stale or incomplete instructions."

新 agent 启动后实际会做的:

  1. cat .multica/issue_context.md ← daemon 在 work_dir 写入的本任务摘要
  2. multica issue get <id> --output json ← 拉 issue 现态
  3. multica issue comment list <id> --output json ← 所有评论(含 Atlas 留下的)
  4. 开始干活
Comment 里到底有多少"上一个 agent 留下的"信息?

Multica 的关键设计:agent 工作产物大量是"反向投射回 issue 评论"的, 而不是停留在 agent 自己的 session 里。

来源怎么写进 issue新 agent 能读到吗
agent post 的进度评论multica issue comment add✅ 在 comments 表
状态切换multica issue status → comments(type='status_change') + activity_logs✅
关联 PRissue_pull_requests 表✅(issue get 输出带)
上传附件attachments 表✅
agent 内部"思考"和工具调用仅在 ~/.claude/.../jsonl❌ 读不到
work_dir 里未 commit 的代码本地文件❌ 跨 runtime 不可见

这就是为什么 Multica 反复教育 agent "在重要节点 post 评论" —— 评论是 agent 间唯一的"信件"。

@mention 与改 assignee 走的 prompt 路径不同吗?

不同。daemon 端按 task.TriggerCommentID 是否为空区分:

  • 改 assignee → buildIssuePrompt, 强调 "读全量历史 mandatory"
  • @mention → task.TriggerCommentID 不空,走 buildCommentPrompt, prompt 显式指出 "这条评论触发了你" + "先读 trigger 所在 thread,再读其它最近活跃 thread", 避免新 agent 啃完整 2000 条评论

Agent 实时中间输出 — task_message 表与 task:message WS 事件 #

一句话: Multica 用 task_message 表存 agent 的中间输出(每个 tool_use / tool_result / text 都是一行),daemon 每条消息一来就批量上报到后端 + WS 广播,前端 AgentLiveCard 订阅 task:message 实时显示。
"agent 干完才落 issue 评论" 只是最终汇报,过程中所有动作早就实时落库 + 推送到前端了。

表 schema

CREATE TABLE task_message (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  task_id   UUID NOT NULL REFERENCES agent_task_queue(id) ON DELETE CASCADE,
  seq       INTEGER NOT NULL,        -- 单调,定 order + 用作 since 游标
  type      TEXT NOT NULL,           -- 'text' | 'tool_use' | 'tool_result' | 'thinking'
  tool      TEXT,                    -- tool 名(Bash / Edit / Read / WebFetch / multica issue ...)
  content   TEXT,                    -- text/thinking 的正文
  input     JSONB,                   -- tool_use 的入参
  output    TEXT,                    -- tool_result 的返回
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_task_message_task_id_seq ON task_message(task_id, seq);

注意:不是 token 级流式,是事件级流式

你跟 ChatGPT 直接聊时有 token 级流式(字一个个蹦); Multica 的实时观察是事件级流式(一段段蹦)。代价是 banner 不会出现"光标一闪一闪打字" 的效果,只会出现"agent 整段整段 + 工具调用块"。 这是平台级 agent 协作产品 vs 单人聊天产品的根本取舍。

daemon 上报失败会重试吗?

不重试。daemon 端 5s 超时 + 失败仅记 taskLog.Debug。

因为 task_message 是过程信号,丢一两条不影响业务正确性。 如果丢了,前端断线重连或刷新 transcript dialog 时通过 GET /tasks/{taskId}/messages?since=<seq> 全量/增量补齐。

"产物 vs 过程"的设计哲学

产物链路(评论 / PR / 状态)严格事务 + 重试; 过程链路(task_message)best-effort + 增量补漏。这一不对称是有意为之的。

脱敏在哪一步做?

入库前。redact.Text / InputMap 在 handler.ReportTaskMessages 里 先 redact 再 insert。

所以 DB 里就是已脱敏版本,后续即使是 admin 直接读表也看不到原文密钥; WS 广播也来自这一份脱敏版,不存在 "DB 安全 / WS 漏密" 的二态问题。 代价是脱敏正则误伤时无法恢复(比如某个 base64 串被当成 token)。

task_message 跨 agent 共享吗?

不共享。换 agent 后新 agent 看不到旧 agent 的 task_message。

这跟 session_id 不跨 agent 是同一种设计 —— 内部观察过程对外不可见,对外信道只有评论 / activity_logs。 这也意味着 transcript dialog 是给"人"看的,不是给"下一个 agent"看的。

Agent 集成层 — daemon 怎么跟 CLI agent 通信 #

一句话: daemon 不是抓终端。它以非交互模式 (-p) + stream-json 协议把 agent CLI 当成标准 stdin/stdout 子进程,逐行解析 JSON。 没有 PTY、没有终端、没有 ANSI 解析、没有正则刮屏。

是怎么"抓"Claude Code CLI 的输出的?

证据 claude.go 的 buildClaudeArgs:

args := []string{
    "-p",                                  // print / 非交互模式
    "--output-format", "stream-json",      // stdout 输出 JSONL
    "--input-format",  "stream-json",      // stdin 也用 JSONL
    "--verbose",
    "--strict-mcp-config",
    "--permission-mode", "bypassPermissions",
    "--disallowedTools", "AskUserQuestion",
}

加上 exec.CommandContext + cmd.StdoutPipe() + bufio.Scanner 逐行读 —— 整个过程没有抓屏。

为什么不抓 TUI

抓屏路线本质上脆弱:ANSI escape sequence 难解析、TUI 布局变化(claude 升级一版就坏)、 进度条/滚动重写历史不可恢复、颜色和文字混编。生产产品撑不起。

stream-json 协议长什么样?

stdout 上每一行都是这种(JSONL):

{"type":"system","session_id":"sess-A","model":"claude-3-5-sonnet"}
{"type":"assistant","message":{"role":"assistant","content":[{"type":"text","text":"好的,我先读 README"}]}}
{"type":"assistant","message":{"role":"assistant","content":[{"type":"tool_use","id":"call-1","name":"Read","input":{"file_path":"/work/README.md"}}]}}
{"type":"user","message":{"role":"user","content":[{"type":"tool_result","tool_use_id":"call-1","content":"# Project\n..."}]}}
{"type":"result","subtype":"success","session_id":"sess-A","duration_ms":12345,"num_turns":4}

要点:

  • message.content 是数组:可同时携带 text + tool_use + thinking 多个 block
  • tool_use 和 tool_result 通过 tool_use_id 配对:相关性靠 ID 不靠顺序
  • system 包裹层带 session_id,result 包裹层带 final summary + total usage
  • 跟 Anthropic Messages API 的 HTTP 协议同构 —— Claude Code 本质是把 API stream 转成 stdout 流
不同 backend 用什么协议?都是 stream-json 吗?

同一个 session_id 是怎么续上上下文的?

核心: Multica 不存任何对话历史。session_id 只是 指向 Claude Code 本地 JSONL 历史文件的句柄。续聊靠 Claude Code CLI 自己读文件, 把整段历史重新喂回给 Anthropic API。

JSONL 位置: ~/.claude/projects/<projectHash>/<session_id>.jsonl

projectHash 由 Claude Code 根据进程的 cwd 算出来。这就是为什么:

  • 必须同时透传 --resume 和 Cwd,cwd 不对找不到 JSONL
  • 跨 runtime 时 session_id 不能续(JSONL 在另一台机器)
  • "续聊"本质 = "把历史 replay 给模型",Anthropic API 本身无状态

代价 — token 累积

每次都把全部历史重新发一次,token 消耗会随对话长度持续放大。 这是为什么 task_usage 表会单独存 cache_read_tokens / cache_write_tokens:Anthropic 的 prompt caching 在续聊场景能命中很多重复前缀,避免重复计费。

resume 失败会怎样?

三层防御:

  1. 中毒黑名单(参见 q-poison-blacklist): 失败原因落在黑名单里直接强制 fresh session
  2. 用户 rerun 永远 fresh:force_fresh_session=true 跳过查询
  3. Resume 后 session_id 可能变:Claude Code 在 corner case 下可能内部决定放弃续聊起新 session, Multica 把后端报回的新 session_id 写回 task,下次跟着新的来

判定 "resume 还没真正建立 session 就失败"(result.SessionID == "")时, daemon 会自动降级到 fresh session 重试一次,而不是把任务彻底标失败。

整条链路的协议栈一共有几层?
链路段物理 / 通道帧编码鉴权
agent → daemonOS pipe\n 分隔UTF-8 JSONL无
daemon 内部Go channel—内存对象无
daemon → serverTCP/TLS + HTTP/1.1 或 2HTTP bodyJSONBearer mul_ / mdt_
server → DBTCP + Postgres wirepgxbinaryDB user/pass
server → 前端TCP/TLS + WebSocketTextMessageJSON握手 cookie/Bearer
server 跨节点RedisPub/SubJSONRedis AUTH

值得知道的设计点

HTTP 上行 + WS 下行的拆分 — daemon 不走 WS 给 server 推,server 永远被动接收,扩容容易。
不在事务里广播 — 写 task_message 后才 publish,WS 上看到的一定是已落库的。
Seq 是 per-task 单调,不是全局 — 父 task 一变 seq 重新从 1 开始,前端按 (task_id, seq) 去重。
