Go 后端结构与核心流程

server/ — Go 1.26 · Chi router · sqlc + pgx/v5 · gorilla/websocket · go-redis/v9

Multica 后端是单一 Go 模块(server/go.mod)同时承担 HTTP API、WebSocket 实时层、本地 daemon 通信、CLI、后台调度等多重职责。本文按"一图一表一段叙事"的方式 拆解九块内容:① 启动装配 → ② 路由与中间件 → ③ Task 三大写入路径 → ④ 状态机 → ⑤ Realtime / DaemonWS / Redis Relay → ⑥ Events Bus 三类监听器 → ⑦ DB 访问层 → ⑧ 后台 goroutine 矩阵 → ⑨ 优雅关闭顺序。

① 启动装配序列 — main.go 是怎么把整个后端"接起来"的

整段 main() 是一份有序的依赖装配清单。任何一个对象都必须等它依赖的对象先创建好, 所以阅读 main.go 等于阅读"后端从冷启动到接受请求的拓扑顺序"。 main.go 第 113-355 行覆盖了从环境变量解析、连接池建立、Hub/Bus 实例化、Redis Relay 选模、监听器注册、HTTP 服务器启动,直到信号优雅关闭的完整流程。

为什么所有 Enqueue 都必须遵循同样的"先广播 WS、再唤醒 daemon"次序

EnqueueTaskForIssue 与 EnqueueChatTask 末尾都明确写着: broadcastTaskEvent(EventTaskQueued) → NotifyTaskEnqueued。 这个顺序不能颠倒。原因是 daemon 接到唤醒后会立刻调 /api/daemon/claim,如果 WS 广播在后,有概率出现: claim 已经把 task 状态改成 dispatched,但浏览器还没收到 task:queued, 于是先看到 task:dispatched 而 list 缓存里没这条记录,UI 上会"凭空跳出 dispatched 卡片"。 把 queued 广播放在前面就能保证两端顺序一致。

EmptyClaim 缓存的失效时机:每次 Enqueue 都会调 EmptyClaim.Bump(runtimeID), 让"该 runtime 没有 queued task"这个推断作废。同时 ClaimTaskForRuntime 在 SELECT 之前会先采样 preSelectVersion,SELECT 完成后 MarkEmpty(runtimeID, preSelectVersion); 如果中途有人 Bump,后续 IsEmpty 检查时版本不匹配就会拒绝命中,从而消除"刚入队的 task 因为缓存还在被忽略"的窗口。

④ Task 全生命周期状态机 — queued → dispatched → running → completed/failed/cancelled

Task 的状态由四方协作者推进:用户(取消)、API server(入队 / 失败标记 / 清扫)、 daemon(claim / start / progress / complete / fail)、sweeper(超时回收)。

Claim 的两种路径

Daemon 既可以"按 agent claim"也可以"按 runtime claim",分别对应两个 service 方法:

  • ClaimTask(agentID):精确匹配 agent + 检查 max_concurrent_tasks + ReconcileAgentStatus。被外部 CLI 用得多。
  • ClaimTaskForRuntime(runtimeID):一台 daemon 上可能有多个 agent,这条路径会先 ReclaimStaleDispatched(回收 90s 内未确认的 dispatched 行),再 EmptyClaim 快速短路,再 ListQueuedClaimCandidatesByRuntime 选优,然后逐个 agent 调 ClaimTask。
"reclaim 90s"是 claim 与 start 之间网络丢包的安全窗口。daemon 调 /claim 拿到 task 后立刻 /start, 但如果 claim 响应丢失(daemon 没收到 200),它会再次 claim 同一台 runtime。此时同一行已经在 dispatched 状态; ReclaimStaleDispatchedTaskForRuntime 会"回收"这一行,前提是它停留在 dispatched ≥ 90s—— 这就是为什么常量定义文档明确写着"必须超过 daemon 客户端 timeout 30s+30s+slack"。

⑤ Realtime Hub + DaemonWS + Redis Relay — 双 WebSocket 通道

Multica 后端运行两个独立 WS Hub:面向浏览器的 realtime.Hub 与面向本地 daemon 的 daemonws.Hub。Redis Relay 仅服务于前者的多节点广播,后者通过 Relay 派生的 RelayNotifier 在跨节点定向唤醒 daemon。

scope-room 与 onFirst/onLast 边界

realtime.Hub 的房间 key 是 {Type, ID}(如 {workspace, ws-uuid}、 {task, task-uuid}、{chat, session-uuid}、{user, user-uuid})。 只要某 scope 从 0 个订阅者变到 1 个,Hub 触发 onFirstSubscriber, ShardedStreamRelay 才启动该分片的 XREADGROUP 循环;反之降到 0 时 onLastSubscriber 停掉循环。这避免一台 API 节点白白消费所有分片消息。

RelayNotifier 是 daemon 跨节点唤醒的关键。Node A 上发生 enqueue,但 daemon 通过 WS 连在 Node B, 此时 Node A 的 daemonHub 并不持有 daemon 句柄;NotifyTaskAvailable 会被 RelayNotifier 包成一帧扔进 daemon-runtime stream, Node B 上的 SetDaemonRuntimeDeliverer(daemonHub) 拣回这帧并通过本地 byRuntime 索引转发给该 daemon。 这是 sharded 模式下 daemon 唤醒能"跨节点"工作的根本原因。

⑥ Events Bus 与三类监听器 — 业务态 → WS / Activity / Inbox 的统一中枢

events/bus.go 是一个同步的进程内 pub/sub。所有 handler 与 service 在写入 DB 后调用 bus.Publish(),再由 main 启动时注册的三组 listener 把事件分发到 WS、activity 表与 inbox 表。

Bus 的核心特性

  • 同步:Publish 在调用方 goroutine 顺序执行所有 handler,无 channel、无重入。
  • 类型路由 + 全局监听:Subscribe(type, h) 只收特定 event,SubscribeAll(h) 收全部。类型 handler 总是先于全局 handler 执行。
  • Panic 隔离:每个 handler 单独 defer recover,一个失败的 listener 不会拖垮其它。
  • Event 字段:Type · WorkspaceID · ActorType · ActorID · Payload + 可选 TaskID / ChatSessionID(用作"提示路由到哪个 scope-room",listener 不必再反序列化 payload)。

⑦ 后台 goroutine 矩阵 — 五条无人值守的常驻例程

除了 HTTP / WS 这种"被请求驱动"的代码路径外,server 进程还运行五条常驻 goroutine。每一条都有自己的触发周期、依赖资源和取消上下文。

goroutine周期取消 ctx主要职责关键阈值
runRuntimeSweeper 30s sweepCtx ① 标 stale 在线 runtime 为 offline · ② 失败 dispatched 超 5min / running 超 2.5h 的 task · ③ 失败 queued 超 2h 的 task(每次 ≤500 行) · ④ GC offline >7d 且无 agent 的 runtime staleThreshold=150s · runningTimeout=2.5h · queuedTTL=2h
heartbeatScheduler.Run 30s 批量 flush sweepCtx 合并 daemon 心跳的 DB 写入,避免热路径上每个心跳触发 UPDATE。Stop 在 srv.Shutdown 之后调用,保证最后一批被刷出 FlushInterval=30s · DBFlushInterval=60s
runAutopilotScheduler 30s autopilotCtx ① 启动时 RecoverLostTriggers(NULL next_run_at) · ② tick 时 ClaimDueScheduleTriggers(SKIP LOCKED) · ③ 调用 DispatchAutopilot · ④ AdvanceTriggerNextRun schedulerInterval=30s
runAutopilotFailureMonitor 由 envFailureMonitorConfig 决定 autopilotCtx 检测连续失败的 autopilot,触发"自动暂停"并通知 owner 窗口 / 阈值由 env 配置
runDBStatsLogger 定期 sweepCtx 把 pgxpool 的 stats(open / idle / acquired)结构化日志输出,便于压测期排查 —

为什么把"取消上下文"分成 sweepCtx / autopilotCtx 两组?

关闭顺序里(参见⑨)autopilotCtx 在收到 SIGINT 时立刻取消,但 sweepCtx 必须等 srv.Shutdown 完成之后才取消。 原因:autopilot 调度可以立即停掉,反正下次启动会 RecoverLostTriggers;但 heartbeatScheduler 必须在所有 daemon 心跳 handler 都返回之后才能 Stop, 否则会丢掉最后 30s 的批量。把这两个生命周期拆开,关闭顺序就清晰且可证明正确。

⑧ DB 访问层 — sqlc 生成 + pgx 连接池 + tx 包装

Multica 不用 ORM,所有 SQL 写在 server/sql/ 下,由 sqlc 编译为 server/pkg/db/generated/ 的强类型 Go 函数。这一节解释 sqlc → Queries → pgxpool 的依赖关系 以及"事务封装"的统一写法。

⑨ 优雅关闭顺序 — 为什么 sweeper 要"晚于" srv.Shutdown 才能停

关闭顺序是 main.go 中最讲究的一段。一个错位就会导致心跳数据丢失、运行中 task 被误判失败。

为什么是这个顺序?

步骤不能放更前是因为不能放更后是因为
autopilotCancel —(可以最早) 不放最前会让 scheduler 在 HTTP 关停期间继续 dispatch,产生"已经在退出却还在写 DB"的 race
srv.Shutdown autopilot 必须先停,否则 daemon /heartbeat 路径会把 sweeper 即将清理的 runtime 的 last_seen_at 又刷新 晚于 sweepCancel 会导致 heartbeat handler 在 sweeper 死后还在调 Schedule,但其实没人 flush
sweepCancel HTTP 必须先全部退出,确保此时不再有人写 batch 晚于 heartbeat.Stop 会让 sweeper 多跑一轮,标错"还活着的"runtime 为 offline
heartbeatScheduler.Stop 必须在 sweepCancel 之后(同一 ctx),让最后一批 ID 被收齐 必须在退出主函数前,否则 60s 心跳数据进不了 DB
metricsServer.Shutdown 必须晚于业务 HTTP,留出最后 3s 给监控系统拉走最新指标 必须在进程退出前(否则没人触发优雅关闭)
relay / redis / pool close —(defer 自然在最后) —
误关一个会怎样:① 如果先 sweepCancel 再 srv.Shutdown,heartbeat handler 会调用一个已经停掉的 scheduler 的 Schedule(), ID 进不了任何缓冲区。② 如果忘了在 srv.Shutdown 后 Stop heartbeat,最后 30s 的心跳全部丢失,sweeper 在下次启动后会把"看上去 90s 没心跳"的 runtime 错判为离线。
