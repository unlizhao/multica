飞书 MeeGo → Multica 迁移可行性调研

文档版本 v0.1 · 基于 multica 当前 main 分支 + MeeGo 公开文档(2024-2025)

这份文档回答三个问题:(1)飞书 MeeGo 与 Multica 各自是什么?(2)它们之间真正的差异在哪里? (3)如果要把公司从 MeeGo 整体搬到 Multica,需要分几步、做哪些事、有哪些坑必须先解决? 结论先放在这里:能迁,但 Multica 的产品形态与 MeeGo 不是同一类工具,需要做"能力补齐 + 工作流改造"两件事才有可能落地, 而非单纯"导出导入"的数据搬运。

来源:飞书 MeeGo / 飞书项目(Meegle) 目标:Multica self-host 能力差距:大 可行性:中-高(需取舍)

执行摘要(给决策者)

三句话结论

  • MeeGo 是低代码 ALM 平台(像 Jira + 飞书集成),Multica 是AI-native 任务平台(像 Linear + Agent)。
  • 核心数据(Issue / Project / 评论 / 附件)可迁;表单 + 流程引擎是 Multica 当前没有的能力,迁移前要先补。
  • 建议分批迁移:先迁研发类工作项(需求、缺陷),再迁项目级管理,最后处理度量与审批类。

预期收益 vs 代价

  • 收益:Agent 一等公民、实时协作体验、私有化部署、源代码可控、年成本可大幅下降。
  • 代价:需补开发表单/流程引擎、自定义字段、度量与报表;迁移期成员需要重新培训。
  • 不建议直接全量切换:先用 6-8 周搭"双轨期",新业务上 Multica,老业务在 MeeGo 收尾。

⚠ 一个关键事实必须先讲清楚

MeeGo 的核心竞争力是"工作项 + 流程节点 + 字段权限"的可配置化组合 —— 公司里 PMO/SQA 已经在 MeeGo 上沉淀了大量流程模板。 Multica 当前没有可配置流程引擎(issue.status 是固定枚举,只能在代码里加新值), 这意味着简单平移会丢掉 MeeGo 的关键能力。要么先在 Multica 上做流程引擎扩展,要么把 MeeGo 流程"扁平化"成 Multica 的 label/project,二选一。

飞书 MeeGo / 飞书项目(Meegle)是什么

飞书 MeeGo(后改名 飞书项目 / Meegle)是字节跳动内部使用、外部商业化的研发协作 + 项目管理平台,定位接近 Jira + Linear + ServiceNow 的子集, 与飞书生态深度绑定(IM 通知、文档、审批、人员体系)。

核心数据模型

Tenant(租户)
  └─ Space(空间)                — 类比 Jira 的 Project
       ├─ WorkItemType(工作项类型)  — "需求 / 缺陷 / 任务" 等可自定义
       │    ├─ Field[]             — 可配置字段(20+ 字段类型,支持公式、关联、附件)
       │    ├─ Workflow / Stateflow— 双流程模式:节点流 / 状态流
       │    │    ├─ Node[]         — 节点 = 阶段,带必填字段、负责人、SLA
       │    │    └─ Transition[]   — 流转规则、驳回、并行/串行
       │    └─ View[]              — 列表 / 看板 / 甘特 / 自定义视图
       ├─ WorkItem 实例
       │    ├─ FieldValue[]        — 一个工作项一行字段值
       │    ├─ Subtask[]
       │    ├─ Comment / Reply
       │    ├─ Attachment
       │    ├─ Activity Log         — 全量审计日志,字段级
       │    └─ Reviews / Approvals  — 评审 / 审批工单
       └─ Role / Permission         — 角色 / 字段级权限 / 节点级操作权限

Multica 是什么(简版,详见 overview)

Multica 是开源、AI-native 的任务管理平台,产品哲学接近 Linear + Cursor 的混合体。最大差异点:

  • Agent 是一等公民:被指派、留评论、报阻塞、改状态。Squad/Skill/Autopilot/Runtime 都围绕"如何让 AI 干活"展开。
  • 核心数据模型固定且收敛:Workspace → Issue/Project/Agent/Squad/Skill/Runtime + 关系(label / pin / subscriber)。
  • 状态机用 Go 枚举写死,通过迁移演化(80+ 次 SQL 迁移),用户不能配置新状态。
  • 极强的实时性(WS + Redis 中继),装机版前端,API 防御边界。
  • 自托管友好,Postgres + Redis + S3 兜底,可单机起步。

读这张表的方式

上半部分(差距大)是 Multica 必须先补的能力;下半部分(反向优势)是 Multica 的核心卖点,迁移之后会获得提升。 所以不是"是否要迁",而是"补齐到什么程度算够、用什么策略补"。

数据模型对比图

关键洞察:MeeGo 让业务管理员通过配置改变系统,Multica 让开发者通过修改代码改变系统。 这种范式差异决定了迁移不是数据搬运,而是"把业务表达从配置语言重新编码到代码语言"。

设计哲学的根本差异

飞书 MeeGo:配置驱动的 ALM 平台

  • 用户群:PMO、SQA、流程工程师 + 研发
  • 价值主张:"把公司流程沉淀到工具里"
  • 扩展方式:管理员在 UI 里点配置 → 立即生效
  • 数据假设:同一个空间里 N 种工作项,每种工作项有独立 schema
  • 权限假设:角色 × 字段 × 节点的高维矩阵,适配大公司
  • AI 形态:辅助工具(写作、规划)

Multica:代码驱动的 AI-native 平台

  • 用户群:研发团队、独立开发者、SaaS 公司
  • 价值主张:"让 AI 和人共用同一个看板"
  • 扩展方式:开发者改源码 + DB migration
  • 数据假设:Issue 是核心,所有差异化用 label / project / metadata 表达
  • 权限假设:工作区 RBAC 3 级,简单清晰
  • AI 形态:Agent 是一等公民,有 profile/被指派/留言

这个差异对迁移意味着什么

公司在 MeeGo 里搭的"5 种工作项 × 12 个流程节点 × 80+ 自定义字段"不是数据,是业务定义。 直接迁过来,你会发现 Multica 没有承载它们的容器。正确做法是先做"流程提炼" —— 把 80% 高频用到的部分提炼出来,在 Multica 上以代码形式实现;余下 20% 长尾流程接受重构或废弃。

迁移决策树:你应该现在迁吗?

五阶段迁移路线图

不要一步到位 —— 用双轨期过渡,中途随时能回退到 MeeGo。每个阶段都有清晰的退出准则与阻塞点。

1
现状盘点 + 立项决策
退出准则:产出迁移 RFC + 高管签字
  • 导出 MeeGo 全部空间清单、每个空间的工作项类型 / 字段 / 流程节点 / 角色
  • 统计:活跃用户数、月新增 issue 数、附件总量、是否依赖飞书审批
  • 用决策树给每个空间打"现在/暂缓/不迁"分级
  • 组建迁移小组:1 名 PM + 1 名 PMO + 2-3 名研发(熟 Go/TS)
  • 产出:迁移可行性 RFC(含本调研引用)、三个月预算、风险登记册
2
Multica 能力补齐(P0 必做)
退出准则:补齐项通过 staging 验收
  • 自定义字段(Custom Field)系统:新建 field_definition + field_value 表,支持 single/multi-select/user/date/text/url/number/relation 8 种基础类型,UI 渲染用 schema-driven form
  • 简化版流程引擎:在 issue 上加 workflow_id,新建 workflow_definition(stages, transitions),UI 上让管理员配置阶段。 不要追平 MeeGo 的全部能力 —— 只做"线性流 + 串行节点 + 必填字段"够覆盖 70% 场景
  • 飞书 Bridge 服务(独立小服务):
    • 飞书 SSO(OAuth2 + 用户映射)
    • 事件 → 飞书 IM 卡片(订阅 multica WS,转发到飞书 webhook)
    • (可选)飞书审批 webhook 接收 + 状态回写到 multica issue
  • 关联类型扩展:补齐 blocks / blocked_by / duplicates / relates_to / split_to
  • 批量导入 API:POST /api/workspaces/:id/issues/bulk-import,带幂等键,允许 dry-run
  • 报表面板补齐:燃尽图 / 累计流图 / 阶段时长(基于已有 events 表派生)
3
小范围试点(1-2 个空间)
退出准则:试点团队 NPS > 0 + 关键流程跑通
  • 挑选研发流程相对标准化、对评审审批依赖弱的小团队(8-15 人)
  • 跑数据迁移 dry-run(见下节),让团队 review 字段映射
  • 实际迁移 → 旧空间设为只读 → 新空间正式使用
  • 双轨 2-4 周:出问题立刻退回 MeeGo
  • 每周收集反馈,锁定 P1 改进列表(尤其是字段渲染、过滤、移动端体验)
4
分批扩展(按业务线滚动迁移)
退出准则:80% 团队完成迁移 · 单团队迁移耗时 < 3 天
  • 按风险/收益矩阵排序业务线:研发类(高收益/低风险)→ 平台类 → 业务线 → 流程重的部门(最后)
  • 每个团队的迁移 SOP:T-7 培训 / T-3 dry-run / T-0 正式迁 / T+1~14 双轨期 / T+15 关闭旧空间
  • 同一时间段最多并行 2-3 个团队迁移,避免 IT 与培训资源被打满
5
收尾 & 长尾归档
退出准则:MeeGo 帐号缩减到只读运维角色
  • 对"不能迁"的长尾流程做评估:重构进 Multica / 沉淀到飞书审批 / 接受废弃
  • 历史数据归档:导出 MeeGo 完整数据到 S3 (JSON + 附件),保留 N 年(合规需要)
  • MeeGo 账号降级到运维角色(只用作历史查询)
  • 沉淀 internal docs:Multica 操作手册、字段命名规范、Agent 使用指南

数据迁移管道(技术细节)

实际跑数据这一步,既要"全量历史"又要"增量同步",还要保证可重放、可回滚。下面是建议的管道架构:

关键设计点

  • 幂等键:在 issue.metadata 上保留 meego_id,loader 用它做 upsert,允许重复跑。
  • 用户映射文件:users.csv (meego_user_key, multica_user_id),人工 review 一次,后续自动用。
  • 字段映射 DSL:用 mapping.yaml 描述 field → 多种目标(field_value / metadata / label / project),支持公式与默认值。
  • 评论时间戳保留:用 created_at_override 字段,避免迁移后所有评论都变成"今天"。需要 multica handler 在迁移用户身份下允许覆盖时间戳。
  • 附件并发上传:大附件直传 S3 + 写入 attachments 表;小附件可以 inline。
  • 双轨期增量:迁移完成后保持 1-2 周增量同步,让用户在两边都能看到新数据,直到信心建立。

关于动 Multica 上游代码

Multica 是开源项目,改动有两种策略:
(A) Fork 维护私有分支:适合公司不打算贡献回上游、愿意自己维护;
(B) 上游 PR + 自建插件层:把字段引擎、流程引擎做成可关闭的功能(feature flag),默认关闭对上游影响小,贡献回 main 后维护成本骤降。
建议优先尝试 (B) —— Multica 当前 80+ 次迁移说明它愿意迭代结构,字段/流程系统有合理化空间。

风险与对策

风险影响对策
用户抗拒:习惯了 MeeGo 的人不愿换高分阶段、双轨期;请试点团队当布道者;Multica 实时性 + Agent 演示制造"新鲜感"
流程能力差距:某流程在 Multica 上确实表达不出来高P2 阶段做"流程精简会",PMO 主持评估;不能精简的留在 MeeGo 或飞书审批
飞书审批依赖:发布单/上线单走飞书审批中用 Multica issue + 飞书 Bridge 双向同步状态,而不是替换审批引擎
历史数据保真度:迁移后字段/状态不一致中把 staging.json 长期保存;在 issue 里加"原 MeeGo 链接",争议时可回查
评论时间戳与作者错乱中handler 支持 override + 用户映射严格 review;迁移工具默认 dry-run
OpenAPI 速率限制低-中extractor 自带退避 + checkpoint;按空间分批拉取
Multica 上游 breaking change中锁定 release tag;改动尽量做成 feature flag;关注上游 release notes
合规 / 数据出境视行业而定self-host 内部部署;附件 S3 用国内云;审计日志保留 N 年
权限模型缩水:从矩阵权限退回 RBAC 3 级中-高P2 决策:接受 / 自建字段权限;同时收紧"敏感字段是否进 Multica"的边界

公司侧待确认问题(决策前必答)

这些问题需要 IT/PMO/研发负责人在决策前给出答案,否则 RFC 写不下去。

  1. 公司在 MeeGo 上有几个空间?每个空间有几种工作项类型?字段总数大致多少?
  2. 是否有强依赖飞书审批的发布/上线/合同流程?如果有,能否在 Multica 上只展示状态、不替换审批?
  3. 有没有必须保留的度量报表?清单是什么(燃尽图、阶段时长、缺陷收敛、其他)?
  4. 用户身份源从哪里来?飞书企业身份?LDAP?自有 IdP?Multica 需要对接哪一个?
  5. 历史数据需要全量迁还是只迁近 N 个月?保留多久?是否需要做合规归档(等保/金融/医疗)?
  6. 谁是这次迁移的 owner(单点责任人)?是否有跨部门协调权?
  7. 预算与时间窗口:本次迁移想在多少周内完成?投入多少人?
  8. 是否考虑把改造贡献回 Multica 上游?(影响 fork 策略)
