Multica · AI-native 任务管理平台

Multiplexed Information and Computing Agent · 致敬 Multics 的"分时复用",这次复用的是人和 AI

像给同事分配任务一样,把 issue 指派给一个 agent —— 它会自己认领、写代码、报阻塞、改状态。 Agent 是一等公民:有 profile、出现在看板、留评论、创建 issue。 支持 Claude Code / Codex / Copilot CLI / Gemini / Cursor / Kimi / Kiro 等十余种 CLI agent, 通过本地 daemon 或云端 runtime 执行。

开源 Go 1.26 + Next.js 16 Electron Expo iOS Self-host 友好 PostgreSQL + pgvector
📐 模块全景与深度解析 → 🗂 目录结构与包用途 → 🧬 数据模型 ER 图 → ⚙️ Go 后端结构与流程 → 🔄 飞书 MeeGo → Multica 迁移调研 →

这是什么

Multica 不是一个"AI 工具",而是一个 人 + AI 协作的任务管理平台。 Agent 和人共用同一个看板;同一个 (agent, issue) 的多轮对话会自动恢复上下文; Skill 系统把历史方案沉淀为可复用能力;Autopilot 让 agent 按规则自动开工。

Agents as Teammates

有 profile,被指派、留评论、创建 issue、报阻塞。

Squads

用 leader agent 路由任务到组,稳定的"团队级别"分配。

Autonomous Execution

WebSocket 实时进度流,任务全生命周期管理。

Reusable Skills

把每次解决方案沉淀为可复用 skill,跨任务复利。

Unified Runtimes

本地 daemon + 云端节点,统一面板,自动发现可用 CLI。

Multi-Workspace

工作区级隔离,各自的 agents、issues、settings。

Go 后端结构

cmd/ 入口
  • server/ — HTTP + WS 服务进程(:8080)
  • multica/ — Cobra CLI(login / daemon / issue / agent ...)
  • migrate/ — 双向迁移工具
internal/handler/ 路由处理
  • 每个领域一个文件:issue.go · agent.go · chat.go · autopilot.go · skill.go · runtime.go ...
  • UUID 解析必走 loader 或 parseUUIDOrBadRequest
internal/realtime/ WebSocket
  • Hub 广播业务事件,失败时通过 Redis Streams 多节点分片中继
  • 模式可配:sharded · dual · legacy
internal/daemonws/ Daemon 通道
  • 与本地 daemon 的双向 WS:心跳调度 / 任务唤醒 / 进度上报
internal/auth · middleware 鉴权
  • JWT + Cookie + PAT(带 Redis 缓存)
  • Daemon Token(mdt_)与用户 Token(mul_)双路
  • CloudFront 私有媒体签名 cookie
  • 基于 Redis 的 IP 限流(RATE_LIMIT_*)
pkg/db/ sqlc
  • queries/*.sql 是真源
  • generated/ 是产物 — 改 SQL 后跑 make sqlc
  • migrations/ 80+ 次迁移,记录业务演化
pkg/agent/ CLI 适配
  • 每个支持的 agent 一个文件:claude · codex · copilot · cursor · gemini · hermes · kimi · kiro · openclaw · opencode · pi
  • 统一 models.go 与 version.go 做能力发现 + 版本兼容
pkg/protocol/ 客户端契约
  • events.go — WebSocket 事件
  • messages.go — daemon ↔ server 消息
  • 前端 core/types/events.ts 必须与之对齐

packages/core 核心

每个领域(issues / inbox / chat / agents / runtimes / projects / squads / autopilots / pins / labels / skills ...)都遵循同一种结构:

issues/ queries.ts # React Query queryOptions + key 工厂 mutations.ts # 乐观更新 ws-updaters.ts # WS 事件 → 缓存失效/补丁 store.ts # Zustand 客户端状态(过滤、视图模式) config/ # 状态枚举、优先级 index.ts # 桶导出
api/client.ts

唯一 ApiClient,所有方法返回经 zod schema 校验的数据。响应字段漂移不会让 UI 白屏。

api/ws-client.ts

自动重连 / 订阅 / 断线广播,服务于 realtime/provider。

platform/core-provider.tsx

一站式装载 ApiClient、stores、QueryClient、WS、I18n。Web/Desktop 都包它。

platform/workspace-storage.ts

当前工作区单例,通过 useSyncExternalStore 接驳。所有 wsId 都来源于此。

auth/store.ts

同时支持 token 模式(Electron)和 cookieAuth 模式(Web HttpOnly)。

realtime/provider.tsx

监听 WS 事件,触发对应 React Query 失效 — 永不直接写 store。

状态管理铁律

关注点归属
服务端数据(issues / agents / inbox / workspaces)TanStack Query · 永不复制到 Zustand
客户端 UI 状态(过滤 / 草稿 / 模态 / 视图模式)Zustand · 只能放在 packages/core/
跨切面装配(workspaceId / navigation)React Context
直接调 api.* 的权限仅 auth store + workspace store
  • ✓WS 事件只触发 invalidate,绝不直接写 store(避免双源真相)
  • ✓workspace 作用域查询必须把 wsId 编进 key,切换工作区时缓存自动切换
  • ✓mutation 默认乐观,失败回滚,settle 时 invalidate
  • ✗Zustand selector 返回新对象/数组会触发无限重渲染 — 选基础值或用 shallow 比较

包边界(违反就是破坏架构)

views/ ──> core/ + ui/ ✓ 允许 core/ ✗ 禁止:react-dom · localStorage(用 StorageAdapter)· process.env · UI 库 ui/ ✗ 禁止:@multica/core · 任何业务逻辑 views/ ✗ 禁止:next/* · react-router-dom · stores(stores 都在 core/) apps/web/platform/ 唯一允许使用 next/navigation 的地方 apps/desktop/.../platform/ 唯一允许使用 react-router-dom 的地方
  • !No-Duplication Rule:web 与 desktop 出现两份相同逻辑必须提到共享包,差异点用 props 槽位(extra / topSlot)
  • !新增页面/功能:组件放 packages/views/<domain>/,web 加 Next.js 路由,desktop 加 router 或 WindowOverlay
  • !新增 hook 需要 wsId:接受 wsId 作为参数(而不是内部读 Context),让它在 WorkspaceIdProvider 之外也能用

API 响应契约(装机版前端的唯一防线)

桌面版客户端会比服务端旧 N 个版本 — 用户在 0.2.x 上访问 0.4.x 后端。每个响应字段都是会漂移的合同。已经发生过 #2143 / #2147 / #2192 三次事故。

  • ✓parse, don't cast:用 parseWithFallback(packages/core/api/schema.ts)+ zod schema + 显式 fallback,失败只 warn,不 throw 进 UI
  • ✗禁止裸 as:任何会被 UI 消费的响应必须过 schema
  • ✓下游全部可选链 + 默认值:每个字段都按可能缺失处理,布尔用 === true 而非真值判断
  • ✓UI 不绑定单字段:按钮 / 指示灯不能依赖单个布尔,需组合多信号
  • ✓Enum 漂移要降级而非崩溃:server-driven 的 switch 必须有 default
  • !新增/修改 endpoint:同 PR 加 schema + 至少一个用畸形响应(缺字段、错类型、null 数组)的测试

业务模块速查

Workspace 工作区

顶层隔离单位,各自的 agents / issues / settings / runtimes / 成员。slug-routed,有保留字列表。

Issue 议题

核心实体:状态机、优先级、assignee(人或 agent)、subscriber、reactions、attachments、labels、子 issue、metadata、关联 PR。

Project 项目

Issue 之上的容器,带 Gantt 视图、资源链接、leader、状态。

Agent 智能体

Profile · 模型 · 思考深度 · 可用 skill · runtime 绑定 · 并发数 · 自定义参数 / 环境变量。从模板创建一键带 skill。

Runtime + Daemon

Runtime = 能跑 agent 的环境(本地 daemon 或云节点)。daemon 自动发现 PATH 上的 CLI,支持云 fleet 代理。

Skill 技能

多文件结构的可复用方案。可从本地 runtime 导入,可被多个 agent 引用。

Autopilot 自动驾驶

定时 / Webhook 触发 agent。带 deliveries 重放、签名密钥轮换。

Chat 对话

跨 issue 的对话会话,挂起任务流,支持 attachments 与 mentions。

Inbox 收件箱

提及 / 状态变更 / 任务完成的通知聚合。批量已读 / 归档。

Squad 小队

leader agent 路由的"团队级"分配:@FrontendTeam 而非单个 agent。

Search 搜索

全工作区命令面板,按 issue / project / agent / skill 检索。

Settings · GitHub · PAT

账户、工作区、成员、Tokens、GitHub App 集成、通知偏好、实验性功能。
